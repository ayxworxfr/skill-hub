# Code Understanding Workflow

## 目标

把“东看一眼西看一眼”的读代码过程，收敛成稳定的理解流程。

## 反偷懒执行规则

- 不只复述文件名、目录名或函数名，必须输出职责、关系和关键路径
- 不在还没读关键入口和主流程前，就草率下结论
- 不闲置搜索和读取能力；要主动定位入口、调用方、数据来源、输出位置
- 不为了显得全面而堆无关细节；要优先收敛主流程和关键证据
- 如果连续两轮仍在同一局部打转，必须切回更高层重新建立结构图

## 适用范围

优先用于以下场景：

- 用户说“梳理代码”
- 用户说“这个模块是干嘛的”
- 用户说“这个函数从哪调用”
- 用户说“数据是怎么流转的”
- 用户说“先看影响范围再改”
- 面对大文件、旧模块、历史包袱代码时需要先建立全局认知

## 工作流

### 1. 先定问题类型

先判断用户更关心哪一种：

- 模块结构
- 调用链
- 数据流
- 影响范围
- 大文件阅读

如果用户没说清，就按“模块结构 -> 调用链 -> 数据流 -> 影响范围”的顺序收敛。

### 2. 先建立整体结构

- 找入口文件、核心类、核心函数
- 区分职责边界：谁负责读取、转换、计算、输出、副作用
- 先回答“这块代码整体在做什么”

### 3. 再追关键路径

- 沿主流程追调用链，不要一开始陷入支线细节
- 标出入口、中转节点、分支点、落点
- 如果是数据问题，重点标出字段从哪里来、在哪里变、最终去哪里

### 4. 最后补边界与风险

- 哪些条件分支最容易出错
- 哪些外部依赖会影响行为
- 哪些调用方或测试会受改动影响
- 哪些部分还没有足够证据，需要继续确认

## 常见分析模式

### 模块结构梳理

- 模块目标是什么
- 核心文件有哪些
- 每个文件的职责是什么
- 建议的阅读顺序是什么

### 调用链分析

- 入口在哪
- 中间经过哪些关键函数/方法
- 分支点在哪
- 最终副作用是什么

### 数据流分析

- 数据来源是什么
- 中间做了哪些转换、过滤、聚合、映射
- 哪些字段最关键
- 数据最终输出到哪里

### 影响范围分析

- 直接改动点在哪里
- 哪些调用方会受影响
- 哪些测试需要回归
- 哪些配置、接口、文件格式可能联动

### 大文件阅读

- 先按职责拆区块
- 找主流程，不按行顺序死读
- 识别重复逻辑、过长函数、隐式耦合
- 先建立地图，再深入局部

## 输出模板

```markdown
## 整体职责

[一句话说明这块代码在做什么]

## 核心结构

- 入口：
- 关键模块/函数：
- 输出/副作用：

## 主流程

1. [步骤 1]
2. [步骤 2]
3. [步骤 3]

## 关键数据流/调用链

- [关键路径 1]
- [关键路径 2]

## 影响范围或风险

- [风险 1]
- [风险 2]
```

## 决策原则

- 先全局，后局部
- 先主流程，后边角分支
- 先基于证据总结，再补推断
- 如果用户只要梳理，就不要直接动代码

## 避免事项

- 不只复述文件名，不解释职责
- 不把调用链和数据流混成一团
- 不在还没看清主流程时就下结论
- 不为了“全面”把无关细节全部堆给用户
