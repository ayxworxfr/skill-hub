# Code Understanding Workflow

## 目标

把读代码从随机浏览变成证据驱动的结构化理解：先建立地图，再追主路径，最后标出风险和影响范围。

## 适用范围

优先用于：

- 梳理模块、解释文件、理解大文件
- 追调用链、数据流、状态流、副作用
- 改代码前评估影响范围
- 面对旧模块、复杂组件、历史包袱

用户只要梳理时，不改代码。

## 强制流程

### 1. 确定理解目标

先判断用户关心哪类问题：

- 模块职责
- 调用链
- 数据流
- 状态流
- 影响范围
- 大文件拆解

没说清时，按“职责 -> 主流程 -> 数据流 -> 影响范围”收敛。

### 2. 建立结构地图

必须找出：

- 入口：谁调用它，或用户从哪里进入
- 核心单元：类、函数、组件、hook、服务、配置
- 输出/副作用：返回值、文件、网络、UI、数据库、状态更新
- 外部依赖：配置、环境变量、第三方库、API、事件

不要一开始陷入边角分支。

### 3. 追主路径

沿主路径列出：

- 输入从哪里来
- 中间经过哪些转换
- 分支条件是什么
- 最终输出到哪里
- 失败路径如何处理

数据流问题必须标字段，调用链问题必须标入口和落点。

### 4. 标影响范围

列出：

- 直接调用方
- 间接受影响模块
- 相关测试
- 相关配置/类型/文档
- 改动风险和未知点

## 检查门

以下情况不得下结论：

- 只读了一个文件，没有入口或调用方
- 只复述函数名，没有解释职责
- 没区分主路径和支线
- 没说明证据来源
- 影响范围没有测试、配置或调用方视角

## 禁止输出模式

| 禁止输出 | 替代动作 |
|---|---|
| “这个模块主要是处理数据” | 说明输入、转换、输出、副作用 |
| “调用比较复杂” | 画出关键调用链和分支点 |
| “影响不大” | 列出检查过的调用方和未检查范围 |
| “大概是这里” | 给出文件/函数证据和不确定点 |
| “代码有点乱” | 按职责拆区块，指出具体耦合点 |

## 输出格式

```markdown
## 整体职责

[一句话说明模块/文件做什么]

## 结构地图

- 入口：
- 核心单元：
- 输出/副作用：
- 外部依赖：

## 主路径

1. ...
2. ...
3. ...

## 数据流或调用链

- ...

## 影响范围

- 直接影响：
- 间接影响：
- 相关验证：
- 未确认：
```
