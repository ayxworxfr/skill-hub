# Bugfix And Debug Workflow

## 目标

禁止症状层补丁。每次修复必须从现象追到可验证根因，再用最小正确改动修掉根因。

## 适用范围

优先用于：

- traceback、测试失败、运行时报错
- 结果不对、UI 行为异常、数据异常
- 本地正常但某环境异常
- 偶发失败、竞态、时序、缓存、配置差异
- 用户问“为什么会这样”“定位一下”“修这个问题”

只分析不改代码时，执行到根因和验证建议为止。

## 强制流程

### 1. 确认现象

记录：

- 实际结果和预期结果
- 触发步骤、输入、环境、时间点
- 稳定复现还是偶发
- 最新错误、日志、截图、失败断言

没有现象证据时，先尝试复现或收集最接近证据。

### 2. 建立证据链

读取并区分：

- 报错点
- 调用方
- 数据来源
- 配置/env/路径/依赖版本
- 最近改动或相关 diff
- 现有测试和边界用例

证据不足时写“不确定”，不要编造 API 或运行状态。

### 3. 追根因

输出内部判断必须区分：

```text
现象：用户看到什么
触发条件：什么输入/环境/状态导致
直接原因：哪段逻辑直接造成
根因：为什么会有这段错误逻辑、错误数据、错误边界或错误假设
验证根因：用什么证据证明它不是猜测
```

如果有多个假设，逐个验证或排除。

### 4. 修复根因

修复规则：

- 优先修错误假设、数据流、边界条件、状态机或配置映射
- 不用默认值吞掉异常
- 不跳过失败分支假装修好
- 不只捕获异常不处理原因
- 不保留失效旧逻辑
- 不顺手重构无关代码

### 5. 验证修复

必须证明：

- 原复现路径不再失败
- 相关边界路径没有破坏
- 新增或现有测试覆盖本次问题
- lint/type/import/build 没有新增明显错误

使用 `verification-workflow` 组织证据。

## 检查门

以下情况必须停下补证据：

- 只知道报错行，不知道输入如何到达
- 只看到异常结果，没有解释为什么产生
- 修复依赖“兜底”“忽略”“跳过”
- 无法复现却直接改核心逻辑
- 验证没有覆盖原失败路径

## 禁止输出模式

| 禁止输出 | 替代动作 |
|---|---|
| “可能是 X” | 给出验证 X 的具体步骤和证据 |
| “加个 try-catch” | 说明异常为什么发生，以及 catch 后如何恢复 |
| “先兜底一下” | 修复导致缺失/异常值的上游原因 |
| “测试通过应该好了” | 指出测试覆盖了原复现路径 |
| “环境问题” | 列出具体环境差异和验证方式 |

## 最终输出

修复类回答必须包含四项：

1. 【根因】一句话说明为什么会错。
2. 【修复】说明完整改动，不用“临时”“先简单”。
3. 【残留】已知差异、未验证项、风险；没有写“无”。
4. 【验证】具体命令、复现路径或对比方式和结果。
