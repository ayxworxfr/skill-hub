# Bugfix And Debug Workflow

## 目标

把“看到异常就直接改”变成稳定流程：

1. 先确认现象
2. 再收集静态和运行时证据
3. 找到直接原因和根因
4. 用最小正确改动修复
5. 立刻验证修复是否生效

## 完整性红线

- 修复必须针对真实问题，不允许用默认值、临时分支、跳过逻辑来掩盖 bug
- 不允许交付“先缓解一下”“先覆盖最常见情况”的半修复版本，除非用户明确只要临时止血方案
- 如果用户要求与生产行为等价，不能只修主路径而忽略已知关键分支、配置差异或数据差异
- 不能为了控制改动面而故意回避明显相关的修复点

## 反偷懒执行规则

- 不在同一报错或同一命令上机械重试；重试前必须说明新依据
- 不在仍可继续收集日志、读取代码、检查配置、验证假设时，直接甩锅给用户或环境
- 不闲置可用工具；能拿证据就不要靠猜
- 不通过反复改动同一小段兜底逻辑制造“好像有进展”的假象
- 不只止住表层报错就停下；默认继续确认根因是否仍存在，并做最小回归验证

## 适用范围

优先用于以下场景：

- Python traceback
- 单测失败
- 逻辑结果不对
- 数据结果异常
- 线上或测试环境特定条件下报错
- 本地正常、某个环境异常
- 偶发失败、条件性失败、日志异常

如果用户明确说“只分析，不要改”，就只执行到根因分析和验证建议这一步。

## 工作流

### 1. 明确现象

- 记录实际结果和预期结果
- 记录输入条件、触发步骤、环境、时间点、相关数据
- 区分是稳定复现还是偶发复现

### 2. 收集证据

- 读取报错位置附近代码
- 查看调用链上下游
- 收集日志、堆栈、配置值、外部依赖返回值、中间数据
- 区分直接证据和推测

### 3. 定位直接原因与根因

至少区分：

- 现象：用户看到什么
- 触发条件：什么情况下出现
- 直接原因：哪段逻辑直接导致问题
- 根因：为什么会有这段错误逻辑、错误数据或环境差异

### 4. 排除伪根因

- 不把结果当原因
- 不把偶然伴随现象当根因
- 如果有多个可疑点，逐个排除
- 如果问题依赖运行时条件，要优先解释环境、配置、时区、数据差异

### 5. 实施修复

- 优先修复真正根因，不只打补丁遮住现象
- 优先最小改动
- 保持原有风格和命名
- 不为“兼容性”保留多余旧逻辑
- 不写 TODO、占位修复
- 不用“兜底默认值”替代真正修复
- 不用“跳过异常分支”假装问题已解决
- 不在明知仍会复现的情况下，把结果描述成修复完成

### 6. 立即验证

- 运行最小相关测试
- 如果没有现成测试，补一个能覆盖本次问题的回归验证
- 如果问题依赖运行时条件，说明复现条件和验证步骤
- 检查是否引入新的 lint 或导入问题

## 输出模板

```markdown
## 现象

[实际结果 vs 预期结果]

## 触发条件

- [条件 1]
- [条件 2]

## 直接原因

[直接导致问题的逻辑]

## 根因

[更底层的真正原因]

## 修复方式

- [修复动作]

## 验证方式

- [测试/复现/日志验证]

## 剩余风险

- [未覆盖点]
```

## 决策原则

- 先证据，后结论
- 先修根因，再处理衍生问题
- 优先缩小复现条件
- 优先检查环境与数据差异
- 证据不足时明确不确定，不编造定论

## 避免事项

- 不看到报错就直接改代码
- 不只修表面症状
- 不只读静态代码就忽略运行时证据
- 不把偶发报错当成纯代码问题
- 不顺手重构无关代码
- 不把“止血”偷换成“修复”
- 不把“少量场景不再报错”偷换成“问题解决”
- 不把“先凑一个能跑版本”当成交付终点

## 触发信号

- “修 bug”
- “定位问题”
- “为什么会这样”
- “运行时报错”
- “这个结果不对”
- “某个环境才会出”
