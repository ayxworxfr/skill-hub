# Large Feature Delivery Workflow

## 目标

把大型需求拆成可独立验证的阶段，防止一次性大改、范围漂移和未验证完成。

## 适用范围

优先用于：

- 跨多个模块、目录、子系统、前后端
- 需要分阶段推进
- 需要长期跟踪进度、风险和验收
- 需要沉淀计划、决策和状态

不要用于：

- 单文件小改
- 纯 bug 定位
- 纯 review
- 用户明确只要分析不改文件

## 强制流程

### 1. 范围收敛

进入实现前必须写清：

- 背景和目标
- 非目标
- 用户/系统入口
- 输入输出边界
- 依赖和配置
- 验收标准
- 交付方式：一次性、分阶段、先设计后实现

关键缺口未补齐时，不进入编码。

### 2. 任务拆解

拆成三层：

```text
Milestone：可对外说明的阶段成果
Work Package：可独立实现和验证的工作包
Action：一次开发循环内完成的动作
```

每个 Work Package 必须包含：

- 目标
- 输入
- 输出
- 依赖
- 风险
- 验证方式
- 完成标准

### 3. 迭代执行

每轮只选择一个主 Work Package：

1. 读取相关上下文。
2. 明确本轮不做什么。
3. 调用对应专项 Skill：前端、配置、测试、调试、重构等。
4. 实现最小完整改动。
5. 运行最小有效验证。
6. 更新进度记录。
7. 汇报完成、验证、风险、下一步。

### 4. 进度记录

如果项目没有用户指定文档，使用 `docs/开发进度规划.md`。

进度记录必须包含：

- 当前目标和非目标
- 里程碑状态
- Work Package 状态
- 最新进展倒序记录
- 验证证据
- 风险和阻塞
- 下一步

状态只允许：

- 未开始
- 进行中
- 已完成
- 阻塞

阻塞必须写原因和解除条件。

### 5. 阶段验收

每个 Milestone 完成时必须：

- 对照验收标准逐项核对
- 关闭或记录风险
- 确认文档状态准确
- 跑相关回归验证
- 明确下一阶段入口

## 检查门

以下情况不得进入实现：

- 没有目标和非目标
- 没有验收标准
- Work Package 不能独立验证
- 依赖链不清
- 风险没有记录

以下情况不得标记完成：

- 没有验证证据
- 只更新口头进度，不更新记录
- 阻塞没有解除条件
- 完成范围与原目标不一致

## 禁止输出模式

| 禁止输出 | 替代动作 |
|---|---|
| “先整体改完” | 拆 Milestone 和 Work Package |
| “进度差不多” | 写事实进展、验证和剩余工作 |
| “后面再补” | 当前阶段不完整就标阻塞或未完成 |
| “风险可控” | 列出风险、影响和解除条件 |
| “这个也顺便做了” | 判断是否越界，必要时拆新工作包 |

## 输出格式

```markdown
## 当前阶段

- Milestone：
- Work Package：
- 本轮目标：

## 本轮完成

- ...

## 验证

- ...

## 风险与阻塞

- ...

## 进度记录

- 已更新：

## 下一步

- ...
```
