# Large Feature Delivery Workflow

## 目标

把“大型需求”从一次性实现，变成可持续推进的工程流程：先收敛范围，再拆解里程碑，再分阶段交付，并持续维护进度文档。

## 适用范围

优先用于以下场景：

- 用户明确提出“大型需求”“多模块改造”“分阶段落地”
- 需求跨前后端、跨多个子系统或多个目录
- 需要持续多轮推进，并且每轮都要有可追踪进展
- 需要把计划、风险、验收、状态写入项目文档

## 非适用范围

以下场景不要使用本 Skill：

- 单文件小改动
- 纯 Bug 定位修复
- 纯代码审查
- 用户明确要求“只分析，不改文件”

## 执行总原则

1. **先定义边界，再动代码**：目标、非目标、验收标准不清晰时，不进入实现。
2. **先拆解再开发**：每个里程碑都必须可独立验证与交付。
3. **每轮都有产出**：要么产出可运行代码，要么产出可执行计划，不做“无效推进”。
4. **持续记录进度**：`docs/开发进度规划.md` 必须跟随每轮变更更新。
5. **风险显式化**：不隐瞒阻塞项，不把未完成内容包装为完成。

## 标准流程

### 阶段 0：范围收敛

必须确认以下内容：

- 业务目标（要解决什么问题）
- 非目标（明确不做什么）
- 输入输出边界（接口、数据、配置、依赖）
- 验收标准（功能、性能、稳定性）
- 交付方式（一次交付 or 分阶段交付）

若存在关键缺口，先提问补齐；不要边写边猜。

### 阶段 1：方案与拆解

将需求拆成 3 层：

1. **里程碑（Milestone）**：可对外汇报的阶段成果
2. **任务包（Work Package）**：每个里程碑下的实现单元
3. **执行项（Action Item）**：可在一次开发循环内完成的具体动作

拆解要求：

- 每个任务包都要写清：输入、输出、依赖、风险、验收方式
- 任务粒度以“半天到两天可完成”为宜
- 依赖链明确，避免后期阻塞

### 阶段 2：迭代开发循环

每轮循环固定执行：

1. 选择一个任务包（只做一个主目标）
2. 阅读上下游代码和配置
3. 实现最小完整改动（不是最小残缺改动）
4. 运行最小有效验证（测试/构建/脚本）
5. 更新 `docs/开发进度规划.md`
6. 汇报“已完成 / 风险 / 下一步”

### 阶段 3：收尾与验收

在里程碑完成时执行：

- 功能验收：对照目标与验收标准逐项核对
- 风险清理：未关闭风险要么解决要么显式记录
- 文档补齐：进度文档状态、决策记录、遗留事项完整
- 回归验证：最小必要回归通过

## 开发进度文档（强制）

### 文件路径

固定使用：`docs/开发进度规划.md`

### 维护规则

- 文件不存在时，立即创建模板
- 每次完成一个任务包后，必须更新
- 状态只允许：`未开始`、`进行中`、`已完成`、`阻塞`
- “阻塞”必须写阻塞原因和解除条件
- 不允许只改百分比，不更新事实内容

### 必填内容

- 当前阶段与目标
- 里程碑列表与状态
- 本周/本轮计划
- 最新进展记录（按时间倒序）
- 风险与阻塞
- 下一步计划

## 回答输出格式

处理大型需求时，优先按以下结构回复用户：

```markdown
## 当前阶段
- 阶段：
- 目标：

## 本轮完成
- ...

## 验证结果
- ...

## 风险与阻塞
- ...

## 已更新文档
- docs/开发进度规划.md

## 下一步
- ...
```

## 反模式（禁止）

- 不拆解任务，直接“大改一坨”
- 用“TODO/后续补充”代替完整实现
- 不更新进度文档，只口头汇报
- 明知阻塞却不记录解除条件
- 把未验证内容写成“已完成”

## 触发信号

- “这是个大型需求”
- “做一个完整方案并推进”
- “分阶段实现”
- “长期跟踪开发进度”
- “在 docs 里维护进度”
