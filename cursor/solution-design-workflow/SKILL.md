# Solution Design Workflow

## 目标

在编码前把问题、约束、可选方案、风险和验收标准讲清楚，避免先写再返工。

## 适用范围

优先用于：

- 用户要“方案”“设计一下”“几个做法”“技术选型”
- 改动存在明显架构取舍
- 需求模糊，直接实现会扩大返工
- 用户明确要求先别写代码

不要用于：

- 用户明确要立刻做一个小而清楚的修改
- 纯 bug 根因排查
- 纯代码 review

## 强制流程

### 1. 定义问题

先写清：

- 目标：要解决什么
- 非目标：明确不做什么
- 输入/输出：数据、接口、文件、UI、命令
- 约束：性能、兼容性、部署、团队习惯、现有架构
- 验收：怎么证明方案落地成功

关键约束缺失时，先问最少必要问题。

### 2. 收集现有依据

方案必须基于证据：

- 当前目录结构和模块边界
- 已有同类实现
- 配置、依赖、构建方式
- 测试和验证能力
- 用户已给出的偏好和禁区

禁止只用通用最佳实践代替项目事实。

### 3. 给出 2-3 个真实方案

每个方案必须包含：

- 核心思路
- 适用场景
- 改动范围
- 优点
- 缺点
- 风险
- 验证方式

不允许把同一个方案换名字凑数。不允许列明显不可落地的假方案。

### 4. 推荐一个方案

推荐必须说明：

- 为什么它最适合当前项目
- 为什么不选其他方案
- 哪些前提变化时推荐会改变
- 下一步实施边界

## 检查门

以下情况不得给最终推荐：

- 没有明确目标和非目标
- 没有读取或引用当前项目事实
- 方案无法比较改动面和风险
- 验收方式不可执行
- 需要用户决策的关键点被隐藏

## 禁止输出模式

| 禁止输出 | 替代动作 |
|---|---|
| “两种方案都可以” | 明确推荐一个，并说明适用前提 |
| “看团队偏好” | 列出影响决策的具体因素 |
| “先做简单版” | 说明这是临时止血还是正式方案 |
| “后续可扩展” | 说明具体扩展点和当前不做的原因 |
| “风险较低” | 写出检查了哪些风险路径 |

## 输出格式

```markdown
## 目标

[一句话目标 + 非目标]

## 方案对比

### 方案 1：[名称]
- 核心思路：
- 改动范围：
- 优点：
- 缺点：
- 风险：
- 验证：

### 方案 2：[名称]
- 核心思路：
- 改动范围：
- 优点：
- 缺点：
- 风险：
- 验证：

## 推荐

- 推荐：
- 原因：
- 不选其他方案的原因：
- 下一步：
```
