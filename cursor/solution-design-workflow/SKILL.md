# Solution Design Workflow

## 目标

把“先写代码再返工”改成先比较方案、再做选择。

## 反偷懒执行规则

- 不把同一方案换几种说法冒充“多个方案”
- 不写明显不可落地、只是凑数的假方案
- 不在已经可以比较改动范围、风险和维护成本时，还停留在空泛表述
- 不拿“先做一个近似版/临时版”当成正式推荐方案，除非用户明确要求临时止血
- 如果连续两轮方案讨论没有收敛，必须明确卡点并缩小决策范围

## 适用范围

优先用于以下场景：

- 用户说“给方案”
- 用户说“有几种做法”
- 用户说“先别写代码”
- 任务存在明显的结构选择或技术取舍
- 改动范围较大，直接实现风险高

## 工作流

### 1. 先定义问题

- 明确目标是什么
- 明确不做什么
- 明确输入、输出、边界条件
- 明确性能、可维护性、兼容性等约束

### 2. 列出 2-3 个可行方案

- 只保留真正可落地的方案
- 不凑数，不写明显劣质的假方案
- 每个方案都要能在当前项目里实施

### 3. 比较关键维度

每个方案都要至少说明：

- 核心思路
- 适用场景
- 优点
- 缺点
- 改动范围
- 风险点

### 4. 给出推荐结论

- 明确推荐哪一个方案
- 说明推荐原因
- 如果需要用户拍板，指出最关键的选择点

## 输出模板

默认使用以下结构：

```markdown
## 目标

[一句话说明要解决的问题]

## 方案 1

- 核心思路：
- 适用场景：
- 优点：
- 缺点：
- 改动范围：
- 风险：

## 方案 2

- 核心思路：
- 适用场景：
- 优点：
- 缺点：
- 改动范围：
- 风险：

## 推荐

- 推荐方案：
- 原因：
- 需要确认：
```

## 决策原则

- 优先选择最容易验证的方案
- 优先选择与现有代码风格最一致的方案
- 优先选择长期维护成本更低的方案
- 如果用户只要方案，就不要提前开始实现

## 避免事项

- 不把实现细节堆成方案对比
- 不写超过 3 个方案导致结论发散
- 不在关键约束未明确时强行拍结论
- 不假装所有方案成本都差不多

## 触发信号

- “方案”
- “设计一下”
- “先分析可选做法”
- “给我两个/三个方案”
- “这个应该怎么设计”
