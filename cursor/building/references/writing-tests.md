# Writing Tests 子模式详解

子模式 = `test`（写测试 / 加回归 / 覆盖 X 行为 / 补单元 / 集成测试）。配合 building/SKILL.md 共用骨架使用。

锚定方法论：TDD 红绿循环（Beck 原始版本）+ Simon Willison "shortest spell for coding agents" + 失败模式表驱动。

## 测试基本原则

测试必须验证行为，不服务于掩盖半成品实现。

- Bug 修复优先写能失败的回归用例
- 重构 / tidy 先确认旧行为有覆盖再改结构
- 失败模式表里的 High 级**先写 RED 测试**（推荐 TDD）

## TDD 红绿节奏（推荐，不强制）

每个 High 级失败模式 / 每个新行为：

1. **红**：写测试 → 跑 → **应该失败**（看到失败信息确认测试是真测了这个东西）
2. **绿**：写最小代码让测试通过
3. **整理**（可选）：tidy 子模式做结构整理；测试保持绿

### 为什么对 AI 重要

- 防 silent fake success：测试先红再绿，证明测试真的在验证目标行为
- 防"修改测试让代码通过"：测试是先于代码存在的目标，不是事后追加的橡皮图章
- 失败信息暴露假设错误：红的时候看到的失败信息往往揭示意图与实现差距

### AI 不能自评测试

测试质量必须**人审一次**，至少抽样：

- 测试断言是否真的在测目标行为（而不是测试桩本身）
- 测试是否能在实现错的时候真的失败
- 边界条件是否被覆盖

人审通过前不算"测试已就绪"。

## 每个测试必须写清

- 输入
- 行为
- 预期输出或副作用
- 失败路径或边界条件

## 聚焦测试规则

- 一条测试验证一个主要行为
- 测试数据小而完整
- 断言外部可观察结果（返回值、文件、异常、调用参数、输出）
- **不复制实现逻辑生成期望值**（手算 / 写死 / 来自 spec）
- 不用宽松断言遮住错误（避免 `assert result is not None` 这类无信息断言）

## 跟随项目模式

读现有测试目录、命名、断言风格、fixture / setup、参数化、mock、临时文件 / env 处理、运行命令。**不引入仓库未采用的测试风格，除非现有方式无法覆盖。**

## 修复 UT 的业务代码边界

修复单元测试失败时，默认只改测试代码、fixture、mock、测试数据或测试环境配置；**不要直接改业务代码来让测试变绿**。

如果排查后确认失败暴露的是业务代码真实缺陷，必须先停下并告知用户：

- 失败测试和关键报错
- 为什么这是业务代码缺陷，而不是测试断言 / fixture / mock 问题
- 需要修改的业务文件和预期行为变化
- 最小验证命令

只有用户明确批准后，才可以修改业务代码。批准前不得把业务代码改动和测试修复混在同一轮输出里。

## 失败模式 → 测试映射

planning 输出的失败模式 ↔ 验证表里 High 级条目，每条对应一个 RED 测试：

```text
失败模式：用户连续点提交 2 次创建 2 条订单
测试：test_idempotent_double_submit
  - 输入：await Promise.all([submit(o), submit(o)])
  - 断言：DB count(订单) === 1
  - 红：实现幂等之前应失败
  - 绿：实现幂等后通过
```

## test 检查门

以下情况不得声称测试有效：

- 测试只有 `assert True` / `assert result is not None`
- 测试只覆盖 happy path，但需求 / 失败模式表有明确失败路径
- 断言复制了实现逻辑
- 为了通过测试删除或放宽关键断言
- 修复 UT 时未获用户批准就修改业务代码
- 测试绿了但没有覆盖本次需求或 bug
- High 级失败模式没对应 RED 测试
- 写完测试直接绿（没看过失败信息 = 没确认测试真在测目标）

## test 禁止输出

| 禁止 | 替代 |
|---|---|
| "补了测试" | 说明测试覆盖的输入、行为和预期 |
| "测试过了" | 给出具体命令和结果 |
| "这个不好测" | 说明不可测原因，并给出替代验证 |
| "先 mock 掉" | 说明 mock 边界和真实行为如何保证 |
| "只测主流程够了" | 对照需求 / 失败模式表列出未覆盖分支 |
| "改了测试让它过" | 反过来：先确认测试断言对，再改代码 |
| "我顺手改业务代码让 UT 过了" | 先说明为什么必须改业务代码，并等用户明确批准 |
| "测试随便写写" | TDD 红绿节奏 + 人审一次 |

## test 输出片段

```markdown
## 测试

- 新增 / 修改：<文件 / 用例 列表>
- 覆盖行为：<输入 → 行为 → 预期>
- 失败模式映射：F-1 → test_X，F-3 → test_Y
- TDD 节奏：
  - 红：跑命令 + 看到的失败信息（截取关键行）
  - 绿：当前命令 + 通过结果
- 命令：
- 结果：
- 未覆盖：
- 业务代码改动：无 / 已获用户批准（附批准内容）
- 人审：是 / 否（否 → 留待用户审）
```
