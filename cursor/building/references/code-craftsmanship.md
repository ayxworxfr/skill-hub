# 封装与抽象纪律

子模式增量约束 = feature（实施新逻辑时同步过）。配合 building/SKILL.md 共用骨架使用。

锚定原则：封装与抽象的目的是降低未来修改成本——欠抽象（堆函数）让改动散，过抽象（造模式）让改动绕。命中信号 ≥2 才动，信号不齐就维持现状。

与 building/SKILL.md §3.1 **R4 (over-engineering)** 互补：R4 阻"过抽象"，本 reference 阻"欠抽象"，双向都是化石。

## 欠抽象信号（≥2 命中 → 该抽）

| # | 信号 | 例 | 抽什么 |
|---|---|---|---|
| **U1** | 同组数据反复一起传 ≥3 个函数 | `(user_id, name, email, age)` 在 4 个函数签名重复 | Model / dataclass |
| **U2** | 函数签名 ≥5 参数 | `f(a, b, c, d, e, ...)` | 参数对象 / Model |
| **U3** | 同类操作复制粘贴 ≥3 处 | 三个文件都做"先校验 → 保存 → 发通知" | 抽函数（DRY） |
| **U4** | 业务实体无对应类型 | 订单只是 `dict[str, Any]`，字段散乱赋值 | Model 类 + 方法 |
| **U5** | 状态用 dict + 字符串 key 传递 | `data["status"]` / `data["payload"]` 在多处用 | dataclass / enum |
| **U6** | 同一逻辑分支被 ≥3 个 if 包裹 | `if type == "A": ... elif type == "B": ...` 多处重复 | 策略模式 / 多态 |

## 过抽象信号（≥1 命中 → 停下退回）

| # | 信号 | 反例 | 替代 |
|---|---|---|---|
| **O1** | 一次性使用的"通用 X" | `UserFactoryStrategyManager` 只有一个实现 | 直接写函数 |
| **O2** | 单实现 interface | `IUserService` 只有 `UserServiceImpl` 一个实现 | 删 interface |
| **O3** | 继承 ≥3 层 | A → B → C → D | 组合替代继承 |
| **O4** | "为未来扩展"抽象 | 当前只有 1 种支付方式，预先抽 `PaymentGateway` | 第 2 种出现时再抽（YAGNI） |
| **O5** | 包装层无新语义 | `UserRepository.save()` 内部只调 `db.save()` | 删包装 |
| **O6** | 模式名塞设计 | 命名含 `Factory` / `Strategy` / `Manager` 但无对应职责 | 重命名为业务动词 |

## 判断顺序

按顺序问，前一条满足再问下一条：

1. **重复了吗？** ≥3 处复制粘贴（U3）→ DRY 抽函数。重复 < 3 处 → 维持
2. **有领域语义吗？** 这组数据 / 状态在业务里有名字（U1 / U4 / U5）→ Model。无名字 → 维持
3. **有多变种吗？** 当前 ≥2 个真实变种（U6）且即将增加 → 模式。只有 1 个变种 → 不要预抽（O4）

逆序错误：先选模式 → 再找数据塞进去 = R4 over-engineering。

## 模式选择速查

只在判断顺序第 3 步命中后用：

| 模式 | 触发信号 | 不该用 |
|---|---|---|
| **策略** | 同一接口 ≥2 个真实实现，调用方按运行时条件选 | 实现只有 1 个 / 实现固定 |
| **工厂** | 创建对象需参数转换 / 选实现类，且 ≥2 处调用 | 只有一处 new / new 无参数转换 |
| **观察者** | 一对多事件通知，监听方独立加减 | 调用关系固定 / 只有 1 个监听方 |
| **适配器** | 外部接口和内部接口形状不一致，且无法改外部 | 接口本来就一致 / 你能改两边 |
| **装饰器** | 同一对象需 ≥2 种可叠加增强 | 增强只有 1 种 / 不需要叠加 |
| **模板方法** | 算法骨架固定，步骤实现因子类不同，且 ≥2 个子类 | 子类只有 1 个 / 骨架不稳定 |

## 中段自检

feature 子模式实施时 checkbox：

- [ ] 函数签名 ≥5 参数 / 同组数据反复传递 → 已对照 U 表
- [ ] 即将用 `Strategy / Factory / Manager / Service` 命名 → 已对照 O 表
- [ ] 抽象前能列出 ≥2 个**当前存在**的真实使用方？无 → 不抽
- [ ] 抽象后改动成本降低还是升高？升高 → 不抽
- [ ] 判断顺序 1 → 2 → 3 走过，没跳到模式

## 检查门

以下情况不得声称"封装合理"：

- 用了模式但只有 1 个真实变种（O4）
- 命名含 `Factory / Strategy / Manager` 但无对应职责（O6）
- 函数 ≥5 参数且数据反复一起传，但没抽 Model（U1 + U2）
- 业务实体到处用 `dict` + 字符串 key，无 dataclass（U4 + U5）
- 抽象层只是 1:1 包装，没消除重复也没加语义（O5）

## 禁止输出

| 禁止 | 替代 |
|---|---|
| "为了未来扩展加个抽象层" | YAGNI：第 2 种实现出现时再抽 |
| "用一下 X 设计模式" | 先列触发信号 ≥1 + 真实变种 ≥2 |
| "直接写个函数实现" / 跳过封装判断 | 过 U / O 信号表，无命中再写函数 |
| "面向未来设计" | 改成"面向当前 2 个真实使用方设计" |
| "这里用工厂模式更优雅" | 给出工厂消除的具体重复 / 转换逻辑 |
| "封装一下" / "抽象一下" 无信号 | 引用具体 U / O 编号 |
