# Config Changes 子模式详解

子模式 = `config`（改 YAML / JSON / env / feature flag / 路径 / 构建配置 / 模型 / provider）。配合主文件共用骨架使用。

## 配置链路（必须建立完整映射）

```text
配置项 → 读取位置 → 解析/默认值 → 传递对象 → 使用分支 → 输出/副作用
```

不要只改配置项名，不检查读取和消费。

## 找全配置来源

定位所有来源：

- 配置文件（YAML / JSON / TOML / INI 等）
- 环境变量
- 命令行参数
- 默认常量
- 构建工具配置
- CI / 部署配置
- 用户本地文件路径

## 同步处理

改动时必须一起更新：

- 字段名和类型
- 默认值和缺省行为
- 路径存在性、目录创建、文件命名
- 下游消费方
- 文档或示例配置
- 测试 fixture

## 至少验证三类场景

- 正常配置路径
- 缺省配置路径
- 错误配置或缺失配置的失败方式

## config 检查门

以下情况不得完成：

- 不知道配置在哪里读取
- 不知道默认值来自哪里
- 改了配置但没改测试 / 示例 / 文档中的同名项
- 没检查路径是否存在或由谁创建
- 没验证缺省场景

## config 禁止输出模式

| 禁止输出 | 替代动作 |
|---|---|
| "改配置就行" | 列出读取方、使用方和默认值 |
| "代码没问题（指 config 改动）" | 说明配置链路如何影响行为 |
| "路径应该存在" | 检查创建方或运行前置条件 |
| "默认值兜底" | 说明缺省行为是否符合产品预期 |
| "只改这里" | 搜索同名配置和测试 fixture |

## config 输出片段

```markdown
## 配置链路

- 来源：
- 读取：
- 默认值：
- 使用：
- 输出：
- 正常配置验证：
- 缺省配置验证：
```
