# 4 种 Skill 骨架

写 new skill 必读。先选骨架再填内容，事半功倍。乱选骨架 = 流程错配，AI 抓不住主线。

## 骨架 A：工作流型

```text
触发 → Step 1 → [产出物] → Step 2 → [产出物] → ... → 交付
                     ↓ 失败
               异常处理 / 回退 / 升级
```

**适用**：步骤有明确先后依赖，前一步产出是后一步输入。

**项目内例子**：`building`、`debugging`、`git-safety`、`refactoring`

**章节模板**：

```markdown
## Core Principle
## 子模式判定（如有）
## 适用范围
## 强制流程
  ### 1. <第一步>
  ### 2. <第二步>
  ### N. 中段自检
  ### N+1. 实施
  ### N+2. 验证
## Final Gate
## 禁止输出模式
## 输出骨架
```

**关键检查**：每步是否有可观察产出？产出能不能被下一步消费？

## 骨架 B：检查清单型

```text
加载清单 → 逐项检查 → 标记状态 (Pass / Warn / Block) → 汇总 → 判断是否通过
```

**适用**：覆盖面优先，每项独立判断。

**项目内例子**：`reviewing-code`、`verifying`、本 skill 的 audit 子模式

**章节模板**：

```markdown
## Core Principle
## 适用范围
## 检查清单
  ### 维度 1: <名称>
  - [ ] 检查项 1.1（带具体动作）
  - [ ] 检查项 1.2
  ### 维度 2: <名称>
  ...
## 状态汇总规则（Pass / Warn / Block 阈值）
## 阻断条件
## Final Gate
## 禁止输出模式
## 输出骨架
```

**关键检查**：每项是否动作化？状态判定是否可量化？汇总规则是否唯一确定？

## 骨架 C：生成器型

```text
接收输入 → 校验输入完整性 → 按模板生成 → 校验输出质量 → 交付
```

**适用**：从少量输入产出标准化文档（PR、commit、文档、报告）。

**项目内例子**：本 skill 的 new 子模式（生成新 SKILL.md）、`git-safety` 的 commit message 部分

**章节模板**：

```markdown
## Core Principle
## 适用范围
## 输入要求
  - 必填字段
  - 可选字段
## 输入完整性校验
  - 缺失必填 → 不允许进入生成阶段
## 生成模板
  <带占位符的标准结构>
## 输出质量自检
  - 占位符是否全部替换
  - 字段是否合规
## Final Gate
## 禁止输出模式
## 输出骨架
```

**关键检查**：模板占位符清晰？输入校验阻断是否生效？

## 骨架 D：分析决策型

```text
收集现象 → 强制多维度分析 → 根因推导 → 方案对比 → 输出结论
              ↑
         禁止输出表拦截浅层回答
```

**适用**：需要深度思考、防止"看起来合理但浅"的回答。

**项目内例子**：`planning`、`debugging` 的根因分析阶段

**章节模板**：

```markdown
## Core Principle
## 子模式判定（如有）
## 适用范围
## 强制流程
  ### 1. 现象/需求收集
  ### 2. 多维度分析（必须 N 个维度都覆盖）
  ### 3. 根因/选项推导
  ### 4. 方案对比（至少 2-3 候选 + trade-off）
  ### 5. 中段自检
  ### 6. 推荐 + 验证计划
## Final Gate
## 禁止输出模式（重点拦截"两种都行"/"看团队偏好"）
## 输出骨架
```

**关键检查**：是否强制多维度？trade-off 是否量化？推荐理由是否落地？

## 选择决策树

```text
任务有先后依赖步骤？
├─ 是 → 工作流型
└─ 否 → 是覆盖面问题（review / audit / verify）？
        ├─ 是 → 检查清单型
        └─ 否 → 输入少 / 输出格式固定？
                ├─ 是 → 生成器型
                └─ 否 → 分析决策型
```

**不知道选哪类时**：默认走工作流型，并显式说明判断依据。后续发现错配再切。

## 混合骨架

部分 skill 跨多类，按主骨架选：

- `building` 主骨架是工作流型，但 test/config 子模式偏检查清单
- `planning` 主骨架是分析决策型，large 子模式偏工作流
- `git-safety` 主骨架是工作流型，含检查清单环节

**规则**：主骨架决定章节结构；子模式按需在 reference 里加自己的小骨架。

## 骨架不该做的事

- 不要把骨架当模板硬填，每章节都要根据 skill 实际任务调整
- 不要 4 类全套（"我都覆盖一下"），只选一个主类
- 不要省略 Core Principle 或 Final Gate（这是三层防御的两端，必有）
- 不要用骨架当借口跳过子模式判定
