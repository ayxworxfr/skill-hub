# 反碎片化（Anti-Fragmentation）：识别打补丁化石

skill 写久了会出现"打补丁感"——不是单点错误，是结构性损耗。本文件给 4 类化石信号 + 检查门 + 修复手法，专供 revise / audit 子模式使用。new 子模式写完主文件前也应过一遍。

## 1. 化石信号（看到一条就要停下重构）

### 信号 A：步骤编号非整数（X.5 / X.0 / Xa）

例：planning Step 2.5（决策驱动变量）、building Step 6.0（多 agent 并行评估）。

含义："事后补的章节硬塞进既有编号体系"的化石，是补丁感最直接的证据。

修复：

- 重新分配整数编号
- 编号冲突时说明"既有流程容量已满" → 拆 reference 而不是叠小数
- 不允许 X.5 / X.0 / Xa 这类非整数后缀

### 信号 B：同一规则在主文件 ≥3 处声明

例：building 的"不许吞异常"同时出现在：

- Core Principle（"覆盖失败路径"原则）
- Step 4 AI 反模式表（"silent fake success" 行）
- 中段自检（一条 checkbox）
- 共用禁止输出表（"应该没问题"行）
- Final Gate（"扫 silent fake success"）

含义：作者每次想加固一条规则就在新位置再写一遍，最终 5 处都在管同一件事。读者注意力被 5 倍稀释，规则反而失效。

修复（DRY）：

- 选一处作为"完整声明"：检测信号 + 阻断条件 + 替代动作三件齐全
- 其他位置改为引用：`见 Step 4 反模式表 #2 silent fake success`
- 中段自检和 Final Gate 引用规则编号 / 关键词，不重复正文
- 三层防御 = 一处声明 + 一处中段引用 + 一处 Final Gate 引用，**不是同一规则写三遍**

### 信号 C：章节维度不正交

例：building 同时存在三个章节：

- "适用范围"（什么时候用 building）
- "子模式判定"（feature / test / config / tidy）
- "任务规模判定（轻量通道）"（Trivial vs 强制流程）

三者维度互相嵌入，读者无法一次性判断"我现在该看哪段"。

含义：作者在不同时点加入了 3 个分类系统但没合并。

修复：

- 列出现有所有"分类章节"
- 标注每个分类的维度名（任务类型 / 任务规模 / 子能力 / 输入形态）
- 维度重叠的合并；维度正交的保留并显式标注
- 单一入口决策表：`if 任务规模 = trivial → 直达；elif 任务类型 = X → 子模式 Y`

### 信号 D：跨 skill 术语断层

例：building 主文件里出现"对齐卡 / 方案卡 / 设计卡 / Walking Skeleton / Decision Drivers / 失败模式表"，但本文件没有任何解释，读者必须先读 planning 才看得懂。

含义：术语来自另一个 skill 的产物，但本 skill 没建立小词典。激活本 skill 时这些字面量直接进入 context，但语义靠"假设用户读过 planning"。

修复：

- 主文件首段建"术语小词典"：每个外来术语一行（一句话定义 + 来源 skill 链接）
- 或者改用本 skill 自有术语，避免跨 skill 名称漂移
- 在 description 里也避免出现外来术语字面量（防误触发判断失准）

### 信号 E：质量门 / 禁止输出过载

例：planning 曾经 H 表 8 + Final Gate 5 + 禁止输出 18 共 31 条检查项。

含义：检查门越多说明主流程越虚——用"加一条检查"代替"把流程写清楚"是逃避，"什么都检查"反而稀释真正重要的几条。流程足够清晰时，质量门自然薄。

修复：

- 上限：H 表 ≤6 / Final Gate ≤3 / 禁止输出 ≤6；超过先想砍，不想加
- 加新检查前先问"主流程哪一步没拦住" → 能改主流程的动词 / 自检 checkbox 就不加检查
- 三表互抢即化石：Final Gate 出现"扫 H1-Hn" / 禁止输出条目能写成"见 §3.1 Hx"——选一处保留

## 2. revise 子模式诊断流程

读现有 SKILL.md 时按以下顺序扫，每条命中给具体修复指令：

| 步骤 | 动作 | 命中即化石 |
|---|---|---|
| 2.1 | grep 步骤标题：`grep '^### [0-9]\+\.[0-9]' SKILL.md` | 信号 A |
| 2.2 | 把每条核心规则的关键词扫出现次数（如 "吞异常"、"silent" 等），>2 即重复 | 信号 B |
| 2.3 | 列出所有 `## ` 章节，标维度；维度交叉即不正交 | 信号 C |
| 2.4 | 列出主文件所有"专有名词"，逐个判断是否本 skill 定义；外来未解释即断层 | 信号 D |

每条化石都要在 revise 输出里给具体修复指令（删哪行 / 合并哪两节 / 加什么链接 / 新增哪个小词典条目）。

## 3. audit 子模式追加 Block 项

audit 时把以下纳入 Block（在 [audit-checklist.md](audit-checklist.md) 之外追加）：

- 步骤编号含小数 / 字母后缀 → 指出哪个 Step，建议合并到哪个整数编号
- 同一规则在主文件 ≥3 处完整声明（不含禁止输出表，禁止输出表本身就是单一来源） → 指出哪几处，建议保留哪一处
- "适用范围 / 子模式 / 任务规模 / 档位"维度交叉且未显式标注 → 指出哪几个章节维度重叠
- 主文件含外来术语未在小词典定义 → 列出术语 + 来源 skill

## 4. 反模式速查

| 反模式 | 信号 | 替代 |
|---|---|---|
| 加规则就找新位置补一遍 | B | 单一来源声明，其他位置引用 |
| 加章节就插小数编号 | A | 重排整数编号或拆 reference |
| 多个分类系统并存 | C | 合并维度或显式正交化 |
| 借用术语不解释 | D | 建小词典或换本 skill 自有术语 |
| "再加一层防御就稳了" | B 的诱因 | 三层防御已是上限：声明 + 自检 + Final Gate；超过就是噪音 |
| "新加一个 Step 6.0 不影响其他编号" | A 的诱因 | 重排整数编号；冲突就拆 reference |
| "适用范围和子模式都列一下更清楚" | C 的诱因 | 合并到决策表，显式标注维度名 |
| 加规则就在质量门多一条 | E | 改主流程动词 / 自检 checkbox；能改流程就不加检查 |

## 5. 修复优先级

发现多条化石时按以下顺序修：

1. 信号 A（步骤编号）→ 最低成本，立刻能改
2. 信号 D（术语断层）→ 加小词典即可
3. 信号 E（质量门过载）→ 删冗余检查 + 重写 Final Gate
4. 信号 C（维度不正交）→ 中等成本，需重排章节
5. 信号 B（规则重复）→ 最高成本，可能要跨章节合并

修信号 C 之前不要修信号 B：维度变了规则也会重排，先改完结构再 DRY。
