# 5 个设计模式（概率空间地形改造）

LLM 不在"遵守"规则，是在概率空间选下一个 token。Skill 设计的本质 = 让正确路径变下坡，让错误路径变高墙。

每个 skill 必须 5 个模式都有落点。任一缺失视为不合格。

## 模式 1：动作化（动词 > 名词）

LLM 看到动词会"启动执行"，看到名词会"表示理解"。"理解"不会改变输出。

### 转化公式

```
"注意 X" → "搜索所有 X 的实例 → 逐个检查 [具体标准] → 列出不合格项"
"关注性能" → "找到所有循环 → 检查嵌套深度 / 是否可缓存 → 列出 O(n²) 以上点"
"代码要简洁" → "统计每个函数行数 → 标记 >30 行为 WARNING、>50 行为 BLOCK"
```

### ✅ 动词起手示例

```markdown
1. 搜索所有 async 函数，检查每一个是否有对应的 try-catch
2. 列出所有外部输入参数，逐个验证是否有 null/undefined 防护
3. 找到所有资源获取点（连接、文件句柄），确认每一个都有释放逻辑
```

### ❌ 名词式声明（禁止）

```markdown
- 代码要有良好的错误处理
- 保持函数简短
- 注意性能
- 写清晰的注释
```

### 自检

每条规则起手词必须是动词：搜索 / 列出 / 检查 / 统计 / 对照 / 找到 / 标记 / 删除 / 替换 / 验证 / 计算。不是动词的全部改写。

## 模式 2：检查门（规则 × 检测 × 阻断）

裸规则 = 建议。规则 + 检测方法 + 阻断条件 = 约束。

### 三要素

| 组件 | 作用 | 缺失后果 |
|---|---|---|
| **规则声明** | 定义标准 | 模型不知道边界 |
| **检测动作** | 强制模型去验证 | 模型"以为"自己遵守了 |
| **阻断条件** | 不通过不能继续 | 模型发现违规但跳过 |

### ✅ 完整检查门

```markdown
## 函数长度检查
1. 统计每个函数的行数（**检测**）
2. 行数 > 30 的函数标记为 WARNING（**规则**）
3. 行数 > 50 的函数标记为 BLOCK（**规则**）

阻断条件：存在任何 BLOCK 级函数时，必须先完成拆分建议，才能继续后续步骤。（**阻断**）
```

### ❌ 裸规则（无效）

```markdown
函数不超过 30 行。
```

### 检查门数量

每个 skill 至少 3 个检查门，覆盖：

- 输入是否合规
- 中段是否偷懒
- 输出是否完整

## 模式 3：三层注意力锚点（防长序列遗忘）

LLM 注意力不均匀：开头权重最高、最近权重次高、中段最容易丢。一条规则在第 200 token 写、到第 2000 token 时影响力已大幅衰减。

### 对策：同一约束出现 3 次

```markdown
## Core Principle（开头声明 — 第一次曝光）

核心约束 X 必须在生成全程持续施加。

## 强制流程

### Step N: 中段自检（中段执行 — 第二次曝光）

- [ ] 我是否已经检查了 X？
- [ ] 我是否在没有 X 的情况下推进了？
→ 后者为是，删除已生成内容，回 Step N-1 重做。

## Final Gate（结尾拦截 — 第三次曝光）

输出前最终确认：
- [ ] X 检查链完整（具体证据可列出）

未通过不得声称完成。
```

### 不是简单重复

- 开头：声明，建立锚点
- 中段：动作化自检，强制回头看
- 结尾：阻断条件，最后拦截

三处的措辞要不同（声明 → 动作 → 检查门），但指向同一约束。

## 模式 4：堵死逃逸（禁止输出表）

LLM 有一种行为模式叫"优雅逃逸"：用看似合理的话术跳过困难步骤。

### 高频逃逸话术

```
"代码整体没有问题"          → 翻译：我不想逐行检查
"建议用户自行验证"          → 翻译：我不确定，但不想承认
"其余部分保持不变"          → 翻译：我懒得输出完整内容
"可能是 X 导致的"            → 翻译：我没验证就猜
"应该没问题"                → 翻译：我没跑命令
"稍后可以扩展"              → 翻译：我现在不想做
```

### 解法：预置为禁止输出

```markdown
## 禁止输出模式

以下表述出现 = 你正在偷懒，必须删除并重做：

| 禁止输出 | 替代动作 |
|---|---|
| "代码整体没问题" | 列出具体检查了哪些路径，每条结论 |
| "建议用户自行测试" | 给出具体测试命令和预期结果 |
| "其余保持不变" | 输出完整内容，或明确"未修改清单" |
| "可能是 X 导致的" | 给出验证是否为 X 的具体步骤 |
```

### 数量要求

每个 skill 至少 5 条禁止输出。覆盖该领域常见偷懒模式。

### 这不是"提醒"

是在概率空间设路障。模型即将生成"代码整体没问题"时，"知道"这条路径被标记错误，概率显著降低。

## 模式 5：渐进披露（控制认知负荷）

人的工作记忆 7±2 项；LLM 注意力窗口虽大，但同样遵循"信息越多 → 单条影响力越低"。

**这是 AI 写 skill 时最常忘的模式**。原因：写作惯性 = 想到啥写啥，不预先分文件 → 写完一长串才"想到"应该拆 → 拆完结构混乱。

### 解法：结构先行（写之前就分配好 references）

下笔写任何 `## ` 章节正文之前，必须先输出文件预算表，详见 SKILL.md Step 1.5。

写作顺序（不是写完再拆）：

1. **文件预算表**：列出所有文件 + 行数估算 + 内容范围
2. **references 骨架先**：每个 reference 文件先写标题 + 章节大纲（可空内容）
3. **主 SKILL.md**：正文中显式 `[X.md](references/X.md)` 链接到对应 reference
4. **回填 references**：最后填充 references 内容

**拒绝的写作模式**：

- ❌ 主文件先写完整 600 行 → 然后看哪些可以拆 → 拆完主文件结构散乱
- ❌ 想到一个新章节就往主文件加 → 没控制总量
- ❌ "先写写看，过 100 行再拆"→ 注意力已被污染，拆不干净

### 策略

正文只放执行指令，背景知识按需加载。

### Token 预算分配

| 内容类型 | 位置 | 理由 |
|---|---|---|
| 触发条件 | YAML description | 必须始终在内存 |
| 执行步骤 + 规则 | SKILL.md 正文 | 触发后加载，核心指令 |
| 参考文档 > 100 行 | references/ | 按需，避免挤占注意力 |
| 代码模板 | assets/ | 用到再读 |
| 自动化脚本 | scripts/ | AI 调用即可 |

### 拆分判定（满足任一即必须拆，不是可选）

- 单一主题 > 100 行细则 → 独立 reference
- 子模式专属知识（不是所有用户都需要）→ 一个子模式一个 reference
- 表格 > 30 行 → 独立 reference
- 长示例（完整代码、模板）→ 独立 reference 或 assets
- 历史决策记录 → 独立 reference

主文件保留（且仅保留）：

- Core Principle
- 子模式判定表（指向 references）
- 共用强制流程
- 中段自检 + Final Gate
- 共用禁止输出
- 输出骨架

### 主文件行数硬阈值

| 行数 | 状态 | 行动 |
|---|---|---|
| ≤ 300 | 健康 | 继续 |
| 300-500 | 警告 | 立即拆，不允许"先写完再说" |
| > 500 | 违规 | Block，必须拆 |

reference 单文件 > 300 行 → 也要再拆。

### 反例

```markdown
# Bad：把所有子模式知识塞进主文件
## design 子模式（150 行）
## large 子模式（180 行）
## deps 子模式（200 行）
→ 主文件 600+ 行，AI 注意力分散
```

```markdown
# Good：主文件 200 行，子模式拆 references
## 子模式判定
| design | references/solution-design.md |
| large  | references/large-feature-delivery.md |
| deps   | references/dependency-selection.md |
→ AI 按子模式按需加载
```

## 应用顺序

写 skill 时按以下顺序套用 5 模式：

1. **动作化**：先把每条规则写成动词起手
2. **检查门**：给关键节点加规则+检测+阻断
3. **三层防御**：把核心约束分布到 Core Principle / 中段自检 / Final Gate
4. **堵死逃逸**：写禁止输出表，列 5+ 条
5. **渐进披露**：> 100 行的内容拆到 references

完成后回头看：每个模式能不能引用到具体行？不能就补。

## 与 Anthropic 规范的关系

- 5 模式是"怎么写得让 AI 真照做"的方法论
- Anthropic 规范是"格式合不合法、能不能被发现"的约束
- 二者并行，缺一不可
