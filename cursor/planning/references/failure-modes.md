# Failure Modes（失败模式预判 + 验证映射）

planning 共用 reference。design / large 子模式在"失败模式预判"步骤必须加载本文件。

锚定方法论：Klein Pre-mortem + 代码级失败模式 + 优先转 RED 测试。

## Core Principle

失败模式预判 = 在写代码前，把"AI 写完后最可能哪里出 bug"写下来，每条配上**可执行验证**（最好是 RED 测试）。

**不允许的偷懒**：

- "应该没问题"（没做 pre-mortem）
- "风险可控"（没列具体失败模式）
- "测试覆盖"（没说什么测试覆盖什么模式）
- "AI 写的时候注意一点"（不是验证，是借口）

## Code-level Pre-mortem（5 步）

### 步骤

1. **设场景**：站在 AI 已经按方案写完代码的时刻，假设这段代码**已经出 bug**或**被回滚**。
2. **写故事**：花 3-5 分钟独立写"失败原因"。每条不允许"代码可能挂"，必须写"在 X 输入下，因为 Y 实现细节，导致 Z 后果"。
3. **覆盖 5 类**：技术 / 边界 / 集成 / 安全 / 兼容（见下表），每类至少思考一次是否命中。
4. **优先级**：标 High / Medium / Low（标准见下）。
5. **映射验证**：每条失败模式对应一个**可执行验证项**（命令 / 测试 / 现象）。High 必须可转 RED 测试。

### 何时做

- design 子模式：选定推荐方案后、产出执行卡前
- large 子模式：每个 Walking Skeleton / 切片启动前
- deps 子模式：候选 ≥2 时对推荐做（防选错）

### 输出量

至少 3 条具体失败模式 + 各自验证项。少于 3 条 = 没认真做。

## 失败模式 5 类（逐项过）

### 1. 技术（Technical）

代码内部逻辑错误：

- 算法 / 状态机 / 时序错
- 异步竞态、Promise 顺序假设错
- 资源泄漏（未释放连接 / 文件 / 定时器 / 订阅）
- 内存 / 性能踩坑（O(n²) 跑万级数据）
- 类型 / null 边界

### 2. 边界（Boundary）

输入 / 状态 / 数据的边界：

- 空 / null / undefined
- 空集合 / 空字符串
- 超大输入
- 负数 / 零 / 极小值
- 非法格式（类型、编码、时区）
- 重复 / 已存在 / 已删除

### 3. 集成（Integration）

跨边界的失败：

- 调用方契约不满足（参数 / 返回类型变了但漏改）
- 上游 / 下游服务超时 / 失败 / 错误码新值
- 数据库事务 / 一致性
- 配置 / 环境变量缺失或新值
- 文件路径 / 权限

### 4. 安全（Security）

代码层威胁（不是合规）：

- 输入校验缺失（注入 / XSS / Path traversal）
- 鉴权 / 授权边界（谁能调）
- 敏感数据日志泄漏
- 新依赖供应链（CVE / 包名幻觉）

### 5. 兼容（Compatibility）

破坏性变更：

- 公开 API 契约破坏（参数 / 返回 / 错误码）
- 数据 schema 不向后兼容
- 配置项重命名 / 删除
- 调用方未同步更新

## 优先级判定

| 级别 | 判定标准 | 处理 |
|---|---|---|
| **High** | 可能导致：崩溃 / 数据丢失 / 安全漏洞 / 公开接口破坏 / 用户可感知错误 | 必须转 RED 测试，写在执行卡里强制覆盖 |
| **Medium** | 可能导致：内部错误 / 性能下降 / 边缘 case 失败 | 至少有 1 个验证（测试 / 手动 / 命令），不强制 RED |
| **Low** | 极小概率 / 影响范围窄 / 已有他处保护 | 记录即可，不强制验证 |

## 失败模式 ↔ 验证映射模板

```text
失败模式：[场景] + [触发原因] + [后果]
级别：High / Medium / Low
验证项：[可执行命令 / 测试 / 现象]
优先转 RED 测试：[Yes / No / N/A]
```

### ✅ 合格示例

```text
失败模式：用户连续点提交按钮 2 次，因为没有 idempotency 保护，DB 创建 2 条订单
级别：High
验证项：测试 await Promise.all([submit(o), submit(o)]); 断言 DB 只有 1 条
优先转 RED 测试：Yes
```

```text
失败模式：解析 JSON 时如果某字段是字符串而非数字，parseInt 返回 NaN，下游计算全是 NaN
级别：Medium
验证项：测试 parse({"qty": "abc"}) 应抛 ValidationError 而非静默返回 NaN
优先转 RED 测试：Yes
```

```text
失败模式：新加的 import "lodash-es" 项目没装，build 失败
级别：High
验证项：grep package.json + npm ls lodash-es 验证已安装；CI 跑 build
优先转 RED 测试：N/A（不是逻辑测试）
```

### ❌ 不合格示例

- "性能可能不够" — 没场景、没触发、没度量
- "可能有 bug" — 不具体
- "需要充分测试" — 没说测什么
- "上线后观察" — 不是失败模式预判

## 检查门

进入推荐前必须确认：

- [ ] 已做 pre-mortem，独立写出 ≥3 条具体失败模式（场景 + 触发 + 后果）
- [ ] 5 类失败模式（技术 / 边界 / 集成 / 安全 / 兼容）已逐项过
- [ ] 每条标了 High / Medium / Low
- [ ] 每条有可执行验证项
- [ ] High 级失败模式优先转 RED 测试（执行卡里写明）
- [ ] 验证不是空话（"加强监控" / "充分测试" 不算）

任一未通过 → 回此步补，禁止前进。

## 禁止输出

| 禁止 | 替代 |
|---|---|
| "风险可控" | 列出每条失败模式 + 级别 + 验证 |
| "上线后再看" | 写成 RED 测试或可执行验证 |
| "应该不会出问题" | 做 pre-mortem，写 ≥3 条失败模式 |
| "测试覆盖一下" | 列具体测试用例 + 输入 + 期望 |
| "加强监控" | 这是 ops，不在编程方案 plan 范围 |
| "AI 写的时候注意" | 写成可执行检查项 + 测试用例 |
| "做好备份" | 这是 ops，不在 plan 范围 |
| "和 X 沟通了" | 写沟通结论 + 决定 |

## 与子模式的关联

- **design**：失败模式表是 trade-off 分析的输入；High ≥ 3 条则候选可能要换
- **large**：每个 Walking Skeleton 切片启动前更新失败模式；High 必须有 RED 测试在切片内
- **deps**：候选评估的"风险信号"对接失败模式（archived / 单维护者 / 未修 CVE / 包名幻觉）

## 与 building skill 的对接

planning 产出的"失败模式 ↔ 验证表"直接喂给 building 作为：

- TDD 红绿节奏的 RED 测试来源
- Final Gate 必检项
- 越界检查依据（如失败模式涉及调用方 → building 必须验证调用方）
