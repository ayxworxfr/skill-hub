# Code Quality（代码级约束）

planning 共用 reference。design / large 子模式在"识别代码级约束"步骤必须加载本文件。

聚焦**代码层面 AI 写完后能验证的约束**。

## 核心原则

代码级约束 = AI 写完代码后，**人或机器可以执行检查**的具体要求。

**不允许的写法**：

- "性能要好" / "保证安全" / "易于维护"（无法验证 = 不存在）
- "符合最佳实践"（哪条？怎么验证？）
- "代码要优雅"（不可执行）

**合格写法**：

- 约束（具体场景）+ 检查方式（命令 / 测试 / 现象）
- 检查方式必须 AI 自己能跑或能写脚本验证

## 6 类约束（逐项过一遍）

### 1. 性能（Performance）

代码层面而非 SLA 层面：

- 算法 / 数据结构复杂度（O(n²) 跑万级数据 → 不行）
- 是否阻塞主线程 / 事件循环 / UI 线程
- 内存占用上限（流式 vs 全量加载）
- I/O 是否批量 / 是否复用连接
- 不必要的重复计算 / 重复渲染

**检查方式示例**：

- 跑基准脚本，输入规模 N，对比改前/改后耗时
- 在事件循环插 marker，确认任务不阻塞
- 看 React DevTools Profiler / Chrome Performance 火焰图
- `time` 命令测量端到端耗时

### 2. 并发安全（Concurrency）

- 共享状态有无竞态（多请求同时改）
- 操作是否幂等（重试不会出问题）
- 锁 / 原子操作 / CAS 是否正确
- 异步顺序假设（Promise.all 顺序、回调嵌套）
- 资源释放（连接、文件、定时器、订阅）

**检查方式示例**：

- 写并发测试（同一接口 100 次并发调用）
- code review 检查每个 await / lock 边界
- 用 race detector（Go `-race`、Node `--no-warnings` 配合工具）
- 注入故障测试（中途 abort、强制超时）

### 3. 可维护（Maintainability）

- 模块边界清晰（一个文件 / 函数职责单一）
- 副作用隔离（纯函数与副作用分层）
- 命名能让下一个 AI / 人 看懂意图
- 可测性（依赖可注入 / 可 mock）
- 注释只写"为什么"，不复述"做了什么"
- 复杂度（圈复杂度 / 嵌套层数 / 文件行数）

**检查方式示例**：

- 写单元测试覆盖关键分支（覆盖率不是目标，能跑断言才是）
- lint 工具（ESLint / Ruff / golangci-lint）
- 圈复杂度工具（radon / lizard）
- 看是否能用 1-2 句话描述每个函数职责

### 4. 可靠（Reliability）

- 错误处理边界明确（哪里 catch、哪里抛）
- 失败模式可观察（不是静默吞错）
- 输入边界（空 / null / undefined / 超大 / 非法）
- 外部依赖失败兜底（超时、重试、降级）
- 状态机 / 状态转换合法性

**检查方式示例**：

- 边界值测试（空数组、null、最大值、负数）
- 故障注入（断网、超时、依赖返回 500）
- 看错误日志是否带足够上下文
- 检查 catch 块是否真的处理而不是 swallow

### 5. 安全（Security）

代码级而非合规级：

- 输入校验（用户输入 / 外部 API 输入）
- SQL / 命令 / Path / XSS 注入
- 鉴权 / 授权边界（谁能调这个接口）
- 敏感数据处理（日志不打印、加密、脱敏）
- 依赖供应链（新引入的包是否有已知 CVE）

**检查方式示例**：

- 用恶意输入测试边界（OWASP 常见 payload）
- 跑 npm audit / pip-audit / cargo audit
- code review 检查每个外部输入入口
- 检查日志输出，搜敏感关键字

### 6. 兼容（Compatibility）

- 公开 API 契约（参数 / 返回 / 错误码）是否破坏
- 数据 schema 变更是否兼容旧数据
- 配置项变更是否兼容旧配置
- 浏览器 / 运行时 / 语言版本兼容
- 已有调用方是否需要同步改

**检查方式示例**：

- grep 调用方 / 类型检查 / 跑现有集成测试
- 类型系统（TypeScript / Go / Rust）能否覆盖
- 用旧数据样本跑新代码
- 看 changelog 类文件 / 类型定义文件 diff

## 命中判定

不是每条都要写。**命中即写，不命中可省**。

| 场景 | 必查命中 |
|---|---|
| UI / 交互 | 性能（阻塞）+ 可靠（边界值）+ 安全（XSS） |
| 后端接口 | 性能（QPS / 延迟）+ 并发 + 可靠 + 安全（鉴权 / 注入） |
| 数据处理 | 性能（大数据）+ 可靠（边界 / 错误）+ 兼容（schema） |
| 公开 SDK / 库 API | 兼容（必）+ 可维护 + 可靠 |
| 重构 / 内部改造 | 可维护 + 兼容（调用方） |
| 引入新依赖 | 安全（CVE）+ 兼容（运行时）+ 可维护（替换成本） |

## 约束写法模板

```text
[类别]：
- 约束：<具体场景，可触发的条件>
- 检查方式：<可执行命令 / 可观察现象 / 测试用例>
```

### ✅ 合格示例

```text
性能：
- 约束：解析 10k 条 JSON 不阻塞 UI（用户能继续滚动）
- 检查方式：在 UI 主线程插 requestAnimationFrame 计数器，处理过程中每 100ms 至少触发 6 次
```

```text
并发：
- 约束：同一订单 ID 并发提交 2 次不会创建 2 条记录
- 检查方式：写测试 promise.all([submit(o), submit(o)])，断言 DB 只有 1 条
```

```text
兼容：
- 约束：现有 GET /api/users 调用方零改动
- 检查方式：grep 调用方 + 跑现有集成测试 npm run test:integration
```

### ❌ 不合格示例

- "性能要好" — 没场景没度量
- "保证线程安全" — 没说哪个共享状态、怎么验证
- "易维护" — 不可执行
- "保证安全" — 没说防什么、怎么测

## 优先级与互斥

约束之间常互斥，必须显式声明优先级：

| 高优先 | 常被牺牲 |
|---|---|
| 性能（极致） | 可维护（缓存 / 优化复杂化）、安全（绕开校验） |
| 安全（强校验） | 性能（每次校验开销） |
| 可靠（多重检查 / 重试） | 性能（写放大）、可维护（兜底分支多） |
| 可维护（清晰分层） | 性能（多层抽象） |

档位差：

- **对齐卡**：通常 1-2 项命中即可，不需要排优先级
- **方案卡**：列 #1/#2 优先约束 + 互斥取舍
- **设计卡**：完整覆盖 + 互斥取舍 + 重评估条件

## 检查门

进入推荐前必须确认：

- [ ] 6 类约束已逐项过
- [ ] 命中项都写了"约束 + 检查方式"
- [ ] 检查方式可执行（命令 / 测试 / 现象），不是描述性
- [ ] 互斥项已显式声明优先级
- [ ] 没用"性能要好"这类不可验证写法

任一未通过 → 回收事实步骤补，禁止前进。

## 禁止输出

| 禁止 | 替代 |
|---|---|
| "性能要好" | 算法复杂度 + 输入规模 + 检查命令 |
| "保证安全" | 列具体威胁（注入 / XSS / 越权） + 测试 payload |
| "易维护" | 模块职责 + 测试覆盖 + lint 规则 |
| "线程安全" | 共享状态点位 + 并发测试 |
| "兼容旧版" | 列调用方 + 集成测试命令 |
| "符合最佳实践" | 哪条最佳实践 + 怎么验证它落地了 |
| "代码优雅" | 不写，改成可执行检查 |

## 输出片段（嵌入到对齐卡 / 方案卡 / 设计卡）

```markdown
## 代码级约束（命中项）

性能：
- 约束：...
- 检查方式：...

并发：
- 约束：...
- 检查方式：...

可维护：
- 约束：...
- 检查方式：...

（其他类别按命中写；不命中省略）

约束互斥与优先级（方案卡 / 设计卡才需要）：
- #1 优先：...
- 牺牲了：...
- 接受理由：...
```

## 与子模式的关联

- **design**：每个候选必须分别评估对 #1/#2 命中约束的满足度，作为 Decision Drivers 之一
- **large**：每个 Story 拆分时必须明确它对哪些命中约束负责
- **deps**：候选库的代码级表现（性能 / 安全 / 兼容）是健康度八维之一
