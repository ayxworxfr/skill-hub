# Solution Design 子模式详解

子模式 = `design`（用户要"方案 / 几种做法 / 设计 / 技术选型（非依赖）/ 架构取舍"）。配合 SKILL.md 共用骨架使用。

锚定方法论：MADR ADR 简化版（Decision Drivers / Considered Options / Decision Outcome）+ Bezos 单/双向门简化为"影响范围分类" + 多候选 + 项目事实优先。

## 核心原则

design 子模式 = **写出 2-3 个项目契合的真实候选** + **可比较的评分** + **明确推荐 + 理由** + **影响范围分类**。

## 6a. 文档权重档位（先选一档）

| 档位 | 适用 | 必备 |
|---|---|---|
| **对齐卡** | 局部决策、单文件、可逆 | 意图 + 单一推荐 + 验证 + 边界 |
| **方案卡** | 跨多文件、含部分公开影响 | 上述 + 2-3 候选 + Decision Drivers + 影响范围 + diff 预算 |
| **设计卡** | 改公开 API / 引入新基础依赖 / 架构级 | 上述 + 调用方 grep + S/B 拆分 + 决策记录段 + Open Questions |

## 7a. 扩展思路（先发散再收敛）

每个候选方向必须解释解决哪个 Goal 或规避哪个失败模式：

- 是否可以**不改代码**（配置 / 流程 / 文档 / 工具替代）？
- 是否能**复用项目已有能力**（grep 同类实现、已有库）？
- 是否能**分阶段交付**（先做最高价值切片）？
- 是否有**更简单的产品路径**绕开复杂工程？
- 是否有现成**三方库 / 平台能力**（命中即升级到 deps 子模式）？
- 是否有**反向方案**（删复杂度、收范围、改约束）？

至少列 3-5 个方向再筛 2-3 个进入候选，避免一开始就锁定。

## 8a. 生成 2-3 个真实候选

每个候选必须填全：

| 字段 | 含义 |
|---|---|
| 名称 | 短语，能区分 |
| 核心思路 | 一段话讲清做什么 |
| 适用前提 | 在什么前提下这是好选择 |
| 改动范围 | 涉及文件 / 函数 / 接口列表（具体路径） |
| 复用项目能力 | 引用 grep 结果 / 已有同类实现 |
| 优点（Pros） | 至少 3 条，对应 Decision Drivers |
| 缺点（Cons） | 至少 3 条，对应 Decision Drivers |
| 影响范围 | 局部 / 跨模块 / 公开接口 |
| diff 预算估计 | 行数 + 文件数量级 |
| 失败模式 | 引用失败模式表的 ID |
| 验证 | 怎么证明这个方案行（优先 RED 测试） |

不许同方案换名字凑数。不许写明显不可落地的假方案。

生态确实只有单一方案时显式说明（找过哪些替代、为什么排除）。

## 9a. Decision Drivers（决策驱动因子）

trade-off 不靠主观感觉。先抽出 3-5 个本次决策最关键的 Driver（来自 Goals + 命中约束 + 项目事实），每个候选按这些 Driver 打分。

### Driver 选取来源

- Goals 中可量化的成功标准
- 已识别的高优先级代码级约束（性能 / 兼容 / 可维护 等）
- 项目硬约束（技术栈、风格、现有架构）
- 影响范围 / diff 预算 / 风险容忍度

### 评分模板

打分用 1-5 或 ✅/➖/❌，重要的是**所有候选用同一组 Driver、同一刻度**。

| Decision Driver | 权重 | 候选 1 | 候选 2 | 候选 3 |
|---|---|---|---|---|
| 改动小（diff 预算） | High | 5 | 3 | 2 |
| 与现有架构契合（项目事实） | High | 5 | 2 | 5 |
| 可维护性 | Medium | 3 | 4 | 5 |
| 性能（命中约束 #1） | Medium | 4 | 5 | 3 |
| 影响范围（局部 > 公开） | Low | 5 | 5 | 1 |
| **加权总分** | — | **4.4** | **3.4** | **3.4** |

加权计算只是辅助，**最终推荐必须给文字理由**，不许只说"分数高"。

## 10a. 影响范围分类（决定 commit 拆法）

每个候选必须显式标影响范围。它直接决定 building 的 commit 拆法和该不该先用户确认。

| 类型 | 含义 | 决策速度 | 文档深度 |
|---|---|---|---|
| **局部** | 单文件 / 函数内部 / 私有 API | 快决策，70% 信息即可 | 对齐卡 |
| **跨模块** | 多文件 / 内部跨模块接口 | 中等深度评估 | 方案卡 |
| **公开接口** | 公开 API / 数据 schema / 调用方众多 / 难替换基础组件 | 慢决策，必须高信息度 + 调用方 grep | 设计卡 |

### 公开接口信号（命中任一即升级到设计卡）

- 改公开 API 契约
- 改用户可见数据 schema 或行为
- 改持久化数据格式
- 选定难以替换的基础组件（DB / MQ / Auth）
- 引入新长期维护成本（自研框架 / 自托管基础设施）

### 调用方 grep（公开接口必填）

设计卡级别，公开接口变更必须列：

- grep 命令 + 结果（具体文件:行号 列表）
- 抽样 ≥3 个调用方读其上下文，确认改动影响
- 是否需要兼容期 / 同步迁移 / 弃用通知

## 11a. S/B 拆分（设计卡）

锚定 Kent Beck Tidy First (2023)：**结构改和行为改不能塞同一个 commit**。

设计卡级别必须拆：

- **S 类**（Structural）：移动 / 抽函数 / 重命名 / 删 dead code / 加 guard clause；不改 user-visible 行为；现有测试保持绿
- **B 类**（Behavioral）：新功能 / 行为变更 / 修 bug；可能加新测试

**顺序固定**：S → 跑测试通过 → B → 跑新测试通过

每类一次 commit（多个 S 或多个 B 也可拆多次）。commit message 用 `tidy:` / `feat:` / `fix:` 区分。

## 12a. AI 反模式预防（写进推荐里）

每个推荐必须显式预防 AI 编码反模式：

- **防 hallucination**：列出 import / API / 库的来源（具体 lockfile / 已有用法）
- **防 silent fake success**：失败路径明确（不允许 try/catch 吞错、不允许 silent default）
- **防 scope creep**：diff 预算 + 不做清单 + Non-Goals
- **防 over-engineering**：拒绝"为未来可扩展"加抽象（除非证据充分）
- **防 inconsistent edit**：调用方 grep（公开接口）
- **防 pattern matching 滥用**：项目事实引用，不照搬通用模板

## design 检查门

进入推荐前必须确认：

- [ ] 文档权重档位已选定且与方案规模匹配
- [ ] 至少 3-5 个方向发散过，筛出 2-3 个真实候选（生态单一时已显式说明）
- [ ] 每个候选必备字段填全（含影响范围 + diff 预算 + 失败模式引用）
- [ ] Decision Drivers 已抽 3-5 个，所有候选同一刻度评分
- [ ] 推荐有文字理由，不是只说"分数高"
- [ ] 影响范围分类已标
- [ ] 公开接口变更（设计卡）含调用方 grep + 抽样上下文
- [ ] 设计卡含 S/B 拆分顺序
- [ ] 失败模式表已写（见 failure-modes.md）

任一未通过不得输出推荐。

## design 禁止输出

| 禁止 | 替代 |
|---|---|
| "方案 A 更优雅" | 写 Decision Driver 评分 + 牺牲了什么 |
| "差不多" / "都行" | 任意两个候选不许同分；说明区分点 |
| "技术上有挑战" | 改成具体失败模式 + 验证（参考 failure-modes） |
| "用 X 业内主流" | 写在本项目上下文为什么主流 = 适合 |
| "性能没问题" | 给代码级约束 + 检查方式 |
| "改一下接口应该没影响" | 列调用方 grep + 抽样上下文（公开接口） |
| "支持未来扩展" | 写具体扩展点 + 当前不做原因 + 重评估条件 |
| "结构和功能一起改" | 设计卡级别拆 S/B 顺序 |
| "AI 写的时候自己看着办" | 写明确 diff 预算 + 验证项 + 不做清单 |

## design 输出片段

### 对齐卡（精简）

```markdown
## 推荐
- 方案：
- 改动范围：<file:func 列表>
- 影响范围：局部
- 验证：<命令 / 测试>
- 边界（不做）：
```

### 方案卡

```markdown
## 候选

### 候选 1：[名称]
- 核心思路：
- 适用前提：
- 改动范围：<file:func>
- 复用项目能力：
- Pros / Cons（≥3 条 each）：
- 影响范围：局部 / 跨模块 / 公开接口
- diff 预算估计：行数 / 文件数
- 失败模式引用：F-1, F-3
- 验证：

### 候选 2 / 3
（同结构）

## Decision Drivers 评分

| Driver | 权重 | 候选 1 | 候选 2 | 候选 3 |
|---|---|---|---|---|
| ... | ... | ... | ... | ... |

## 推荐
- 选：[候选 X]
- 逐 Driver 理由：
- 不选其他原因：
- 影响范围：
- diff 预算：
- 下一步实施边界（→ building 任务契约）：
- 重评估条件：
```

### 设计卡（额外段）

```markdown
## 调用方 grep（公开接口变更）
- grep 命令：
- 结果：
  - <file>:<line>
  - ...
- 抽样上下文：

## S/B 拆分顺序

S 类（结构改，不改测试）：
1. ...
2. ...

B 类（行为改）：
1. ...
2. ...

顺序：S1 → S2 → 测试 → B1 → 测试 → B2 → 测试

## 决策记录
- 上下文：
- Drivers：
- 决策结果：
- 后果：
- 重评估条件：

## Open Questions
- ...
```
