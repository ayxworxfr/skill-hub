# Large Feature Delivery 子模式详解

子模式 = `large`（跨多文件 / 多阶段 / 重构迁移 / 长链路改造）。配合 SKILL.md 共用骨架使用。

锚定方法论：Cockburn Walking Skeleton + Jeff Patton 垂直切片 + Kent Beck Tidy First S/B 拆分。

## 核心原则

large 子模式 = **先打通最薄端到端骨架** + **每片切片端到端可验证** + **每片有 diff 预算 + 失败模式 + 验证** + **结构改与行为改分开 commit**。

每个切片独立可交付、可验证、可回退到上一切片。AI 一次只做一片，不允许"先把基础做了再说"。

进入切片清单前置条件：SKILL.md Step 2.5 决策驱动变量已 asked 或 inferred-with-evidence（典型 driver：灰度需求 / 是否阻塞他人 / 上线窗口 / 切片粒度偏好）。

## 1. Walking Skeleton（第一刀）

由 Alistair Cockburn 提出。先做最薄的端到端打通，再切切片。

### 是什么

Walking Skeleton = 系统最小完整端到端：从入口到产出，每个环节都用最简实现，连成可运行链路。

不是 MVP，比 MVP 还小。问题不是"用户需要什么最小功能"，而是"端到端跑通最少需要什么"。

### 为什么先做骨架

- 提前暴露**集成风险**（接口契约 / 模块边界 / 数据流）
- 后续切片都基于已可运行的骨架长肉
- 避免 AI"组件全做完最后联调爆雷"
- 第一刀本身就是验证项目事实是否准确的最快方式

### 强制要求

每个 large 计划必须有显式的 Walking Skeleton 切片，且**必填**：

| 字段 | 含义 |
|---|---|
| 范围 | 端到端最薄链路（具体文件 / 函数列表） |
| 占位策略 | 哪些用 stub / 写死 / 空实现，且**显式标注**（不允许悄悄留 TODO） |
| diff 预算 | 文件数 + 行数量级 |
| 验证 | 端到端跑通命令 / RED 测试 |
| 完成标志 | 跑什么命令出什么结果 = 第一刀完成 |

骨架完成前**禁止做边缘场景 / 边界优化 / 性能调优**。

## 2. 垂直切片（不是按层切）

切片必须**端到端**，覆盖入口 → 处理 → 产出全链路一小段。

### 反例（横切，错）

- 切片 1：建表 + schema 迁移
- 切片 2：写后端逻辑
- 切片 3：写前端 / CLI
- 切片 4：联调

→ 每个切片自身不可验证、不可独立回退。AI 做完切片 1 没法证明对错。

### 正例（纵切，对）

- 切片 0（Walking Skeleton）：单一 happy path 端到端跑通
- 切片 1：覆盖核心场景 A 的边界（含 schema、逻辑、入口、测试）
- 切片 2：覆盖场景 B
- 切片 3：覆盖错误路径与失败模式

每个切片完成后跑测试都绿。

### 切片技巧（先选一种切法再切）

| 切法 | 示例 |
|---|---|
| 按场景 | 先核心场景，后边缘 |
| 按数据复杂度 | 先简单结构，再嵌套 / 批量 |
| 按规则覆盖 | 先 happy path，再边界值 / 失败路径 |
| 按操作 | 先读，再写，再删除 / 撤销 |
| 按调用方 | 先一个调用方接入，再扩到其他 |

不允许"按层切"作为本能反应。

## 3. 切片必备字段

每个切片（含 Walking Skeleton）必须填全：

| 字段 | 含义 |
|---|---|
| ID | S-0 / S-1 / S-2 ... |
| 名称 | 短语描述 |
| 范围 | 涉及文件 / 函数 / 接口（具体路径） |
| 入口前提 | 上一切片完成的具体标志（命令 / 测试） |
| 完成标志 | 本切片完成的具体标志（命令 / 测试 / 现象） |
| diff 预算 | 文件数 + 行数量级 |
| 命中约束 | 引用代码级约束表（性能 / 并发 / ...） |
| 失败模式 | 引用失败模式表 ID（F-1 / F-2 ...） |
| 验证 | 优先 RED 测试，列具体用例 |
| 不做清单 | 本切片显式不做的相邻事项（防 scope creep） |

字段缺一不允许进入下一切片。

## 4. 切片内部 S/B 拆分

每个切片内部必须拆 S 类（Structural，不改 user-visible 行为）+ B 类（Behavioral，新功能 / 行为变更 / 修 bug）。

**顺序固定**：S → 跑测试通过 → B → 跑新测试通过。每类一次 commit。

切片内 S/B 同时存在时必须显式列出顺序（顺序由本卡指定，building 按卡执行）。

## 5. 切片间衔接

切片之间不允许"半成品状态"：

- 上一切片**必须留下绿测试 + 可运行端到端**
- 下一切片入口前提引用上一切片的完成标志
- 任何切片中断时，repo 处于可编译可运行状态
- 中间切片已暴露未预期问题 → **回到 design 重评估**，不强行往下推

进入下一切片前必须确认：

- [ ] 上一切片完成标志达成（命令通过 / 测试绿）
- [ ] 上一切片暴露的新失败模式已加入失败模式表
- [ ] 下一切片字段已更新（特别是依赖关系）
- [ ] repo 当前可编译可运行

## 6. 进度记录

如用户未指定文档路径，使用 `docs/开发进度规划.md`。

记录最少包含：

- 当前切片 ID + 状态（未开始 / 进行中 / 已完成 / 阻塞）
- Walking Skeleton 状态
- 切片清单（含每片字段）
- 最新验证证据（命令 / 测试结果）
- 阻塞项（原因 + 解除条件）
- 下一切片入口前提是否满足

阻塞项必须具体，不写"在排查"。

## large 专属禁止输出

主文"共用禁止输出"已覆盖通用项；以下是 large 子模式补充：

| 禁止 | 替代 |
|---|---|
| "先整体框架搭起来" | 第一刀做 Walking Skeleton（端到端最薄链路） |
| "先建表，再写后端，再写 UI" | 改成垂直切片 |
| "切片 1 做基础设施" | 切片 1 必须端到端可验证 |
| "最后一起联调" | 每切片含联调，不允许尾部集中 |
| "切片之间留 TODO" | 上切片留绿测试 + 可运行；TODO 须落到具体切片 |
| "切片 N 为切片 N+1 预留扩展点" | 删，N+1 时再加 |

## 切片产出消费方

每个切片产出的字段直接作为下游实施任务的契约：执行方按字段硬约束执行，触及未列字段 / 文件 / Non-Goals → 停下回到本 skill 重评估。
