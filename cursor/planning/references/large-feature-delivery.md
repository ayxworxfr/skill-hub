# Large Feature Delivery 子模式详解

子模式 = `large`（跨多文件 / 多阶段 / 重构迁移 / 长链路改造）。配合 SKILL.md 共用骨架使用。

锚定方法论：Cockburn Walking Skeleton + Jeff Patton 垂直切片 + Kent Beck Tidy First S/B 拆分。

## 核心原则

large 子模式 = **先打通最薄端到端骨架** + **每片切片端到端可验证** + **每片有 diff 预算 + 失败模式 + 验证** + **结构改与行为改分开 commit**。

每个切片独立可交付、可验证、可回退到上一切片。AI 一次只做一片，不允许"先把基础做了再说"。

## 6c. 范围收敛

进入拆解前必须写清（沿用 SKILL.md 第 3 步成果，不重写）：

- Job-to-be-Done + Goals + Non-Goals
- 项目事实引用（grep / read 的具体路径）
- 命中代码级约束
- 改公开接口的部分附调用方 grep
- 总 diff 预算（量级，如 10-30 文件 / 500-2000 行）

无以上前置，不进入拆解。

## 7c. Walking Skeleton（第一刀）

由 Alistair Cockburn 提出。先做最薄的端到端打通，再切切片。

### 是什么

Walking Skeleton = 系统最小完整端到端：从入口到产出，每个环节都用最简实现，连成可运行链路。

不是 MVP，比 MVP 还小。问题不是"用户需要什么最小功能"，而是"端到端跑通最少需要什么"。

### 为什么先做骨架

- 提前暴露**集成风险**（接口契约 / 模块边界 / 数据流）
- 后续切片都基于已可运行的骨架长肉
- 避免 AI"组件全做完最后联调爆雷"
- 第一刀本身就是验证项目事实是否准确的最快方式

### 强制要求

每个 large 计划必须有显式的 Walking Skeleton 切片，且**必填**：

| 字段 | 含义 |
|---|---|
| 范围 | 端到端最薄链路（具体文件 / 函数列表） |
| 占位策略 | 哪些用 stub / 写死 / 空实现，且**显式标注**（不允许悄悄留 TODO） |
| diff 预算 | 文件数 + 行数量级 |
| 验证 | 端到端跑通命令 / RED 测试 |
| 完成标志 | 跑什么命令出什么结果 = 第一刀完成 |

骨架完成前**禁止做边缘场景 / 边界优化 / 性能调优**。

## 8c. 垂直切片（不是按层切）

切片必须**端到端**，覆盖入口 → 处理 → 产出全链路一小段。

### 反例（横切，错）

- 切片 1：建表 + schema 迁移
- 切片 2：写后端逻辑
- 切片 3：写前端 / CLI
- 切片 4：联调

→ 每个切片自身不可验证、不可独立回退。AI 做完切片 1 没法证明对错。

### 正例（纵切，对）

- 切片 0（Walking Skeleton）：单一 happy path 端到端跑通
- 切片 1：覆盖核心场景 A 的边界（含 schema、逻辑、入口、测试）
- 切片 2：覆盖场景 B
- 切片 3：覆盖错误路径与失败模式

每个切片完成后跑测试都绿。

### 切片技巧（先选一种切法再切）

| 切法 | 示例 |
|---|---|
| 按场景 | 先核心场景，后边缘 |
| 按数据复杂度 | 先简单结构，再嵌套 / 批量 |
| 按规则覆盖 | 先 happy path，再边界值 / 失败路径 |
| 按操作 | 先读，再写，再删除 / 撤销 |
| 按调用方 | 先一个调用方接入，再扩到其他 |

不允许"按层切"作为本能反应。

## 9c. 切片必备字段

每个切片（含 Walking Skeleton）必须填全：

| 字段 | 含义 |
|---|---|
| ID | S-0 / S-1 / S-2 ... |
| 名称 | 短语描述 |
| 范围 | 涉及文件 / 函数 / 接口（具体路径） |
| 入口前提 | 上一切片完成的具体标志（命令 / 测试） |
| 完成标志 | 本切片完成的具体标志（命令 / 测试 / 现象） |
| diff 预算 | 文件数 + 行数量级 |
| 命中约束 | 引用代码级约束表（性能 / 并发 / ...） |
| 失败模式 | 引用失败模式表 ID（F-1 / F-2 ...） |
| 验证 | 优先 RED 测试，列具体用例 |
| 不做清单 | 本切片显式不做的相邻事项（防 scope creep） |

字段缺一不允许进入下一切片。

## 10c. S/B 拆分（每个切片内部）

锚定 Kent Beck Tidy First (2023)：**结构改和行为改不能塞同一个 commit**。

每个切片内部必须拆：

- **S 类**（Structural）：移动 / 抽函数 / 重命名 / 删 dead code / 加 guard clause；不改 user-visible 行为；现有测试保持绿
- **B 类**（Behavioral）：新功能 / 行为变更 / 修 bug；可能加新测试

**顺序固定**：S → 跑测试通过 → B → 跑新测试通过

每类一次 commit（多个 S 或 B 也可拆多次）。commit message 用 `tidy:` / `feat:` / `fix:` 区分。

切片内 S/B 同时存在时必须显式列出顺序。

## 11c. 切片间衔接

切片之间不允许"半成品状态"：

- 上一切片**必须留下绿测试 + 可运行端到端**
- 下一切片入口前提引用上一切片的完成标志
- 任何切片中断时，repo 处于可编译可运行状态
- 中间切片已暴露未预期问题 → **回到 design 重评估**，不强行往下推

## 12c. AI 反模式预防（写进切片清单）

每个切片必须显式预防：

- **防 scope creep**：每片 diff 预算 + 不做清单
- **防 silent fake success**：失败路径切片显式覆盖
- **防 over-engineering**：拒绝"为后面切片预留扩展点"，下一切片再加
- **防 inconsistent edit**：跨文件改动列调用方 grep
- **防 happy path only**：失败模式至少有一片显式覆盖

## 13c. 进度记录

如用户未指定文档路径，使用 `docs/开发进度规划.md`。

记录最少包含：

- 当前切片 ID + 状态（未开始 / 进行中 / 已完成 / 阻塞）
- Walking Skeleton 状态
- 切片清单（含每片字段）
- 最新验证证据（命令 / 测试结果）
- 阻塞项（原因 + 解除条件）
- 下一切片入口前提是否满足

阻塞项必须具体，不写"在排查"。

## large 检查门

进入第一切片前必须确认：

- [ ] 范围收敛字段齐全（沿用 SKILL.md 第 3 步成果）
- [ ] 总 diff 预算已写
- [ ] Walking Skeleton 切片必备字段全填
- [ ] 切片清单按垂直切片切（非按层）
- [ ] 每个切片必备字段全填
- [ ] 每个切片 S/B 拆分顺序明确
- [ ] 失败模式表已写（见 failure-modes.md）
- [ ] 改公开接口的切片含调用方 grep（见 solution-design.md 设计卡）

任一未通过不得开始第一切片。

进入下一切片前必须确认：

- [ ] 上一切片完成标志达成（命令通过 / 测试绿）
- [ ] 上一切片暴露的新失败模式已加入失败模式表
- [ ] 下一切片字段已更新（特别是依赖关系）
- [ ] repo 当前可编译可运行

## large 禁止输出

| 禁止 | 替代 |
|---|---|
| "先整体框架搭起来" | 第一刀做 Walking Skeleton（端到端最薄链路） |
| "先建表，再写后端，再写 UI" | 改成垂直切片 |
| "切片 1 做基础设施" | 切片 1 必须端到端可验证 |
| "最后一起联调" | 每切片含联调，不允许尾部集中 |
| "切片之间留 TODO" | 上切片留绿测试 + 可运行；TODO 须落到具体切片 |
| "切片 N 为切片 N+1 预留扩展点" | 删，N+1 时再加 |
| "差不多了" | 列完成标志命令 + 结果 |
| "结构和行为一起改" | 切片内拆 S/B 顺序 |
| "AI 写的时候自己控制范围" | diff 预算 + 不做清单 + 调用方 grep |

- docs/开发进度规划.md 或用户指定

## 与 building skill 的对接

每个切片产出后直接喂给 building 作为任务契约：

- 切片范围 → building 任务契约的"目标 / 输入 / 输出"
- diff 预算 + 不做清单 → building 越界即停依据
- 失败模式表 → building TDD 红绿节奏的 RED 测试来源
- 完成标志 → building 完成判定
