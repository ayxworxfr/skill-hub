# Dependency Selection 子模式详解

子模式 = `deps`（找三方库 / SDK / npm / PyPI / Maven / Cargo 包 / 替换依赖）。配合 SKILL.md 共用骨架使用。

评估维度详解、各语言查证命令、OpenSSF / 供应链工具：见 [dependency-evaluation.md](dependency-evaluation.md)。

进入候选健康度评估前置条件：SKILL.md Step 2.5 决策驱动变量已 asked 或 inferred-with-evidence。deps 典型 driver 速查见 [clarifying-questions.md](clarifying-questions.md) `## 各子模式 driver 速查`：平台 / 用户量级 / 团队规模与技术栈 / 已有依赖 / 许可证约束 / 是否跨平台导出。

driver 未解决禁止进入健康度评估或给最终推荐。

## 1. 判断是否需要依赖

回答：

- 需求是否通用，生态可能已有方案？
- 自研是否容易踩安全 / 兼容 / 性能 / 边界坑？
- 依赖成本：包体、供应链、安全、许可证、维护、学习、迁移
- 项目是否已有等价依赖或内部封装

结论必须是四选一：使用现有 / 新增三方 / 自研 / 暂不决策（缺信息）。

## 2. 联网查真实候选

需要新增或替换时，必须联网查：

- 官方包管理器：npm、PyPI、Maven Central、crates.io、pkg.go.dev
- GitHub / GitLab 仓库
- 官方文档、生态指南、框架推荐
- 社区讨论、issue、迁移建议

默认比较 2-3 个候选；只有单一主流方案时说明依据。具体语言查证命令见 [dependency-evaluation.md](dependency-evaluation.md) `## 各语言查证命令`。

## 3. 健康度评估

每个候选必须按 evaluation.md 8 个维度过一遍：功能匹配 / 维护活跃度 / 社区采用 / 维护者治理 / 安全供应链 / 许可证 / 工程质量 / 项目适配。

## 4. 项目适配

推荐前对照当前项目：

- 技术栈和运行环境是否匹配
- 是否与已有依赖重复
- 是否破坏构建、类型、测试、部署、包体目标
- 是否需要封装内部 adapter / UI layer
- 是否有最小接入路径和退出路径

## 5. 接入与验证

deps 推荐输出必须额外包含：

- 安装方式（具体命令）
- 最小接入示例
- 验证命令或检查路径
- 后续监控方式

## 风险默认标红

命中任一立即标红，不靠总分掩盖：

- 最近 12 个月无 release 且无明显维护活动
- 仓库 archived / deprecated / 废弃
- 单维护者且 issue/PR 长期堆积
- 无 LICENSE
- GPL/AGPL 等可能影响项目分发，且未获用户确认
- 未修 critical/high CVE
- 安装脚本 / 依赖树 / 供应链行为异常
- 文档稀薄、API 不稳定

各维度更细的风险信号见 [dependency-evaluation.md](dependency-evaluation.md)。

## 风险标签

- `低风险`：维护活跃、许可证清楚、安全无明显问题、项目适配成本低
- `中风险`：部分指标不完整，但有可接受的缓解方案
- `高风险`：维护停滞、许可证不明、未修漏洞、供应链异常、强侵入项目架构
- `阻断`：无 LICENSE、critical/high CVE 未修、疑似仿冒包、与项目法律 / 安全要求冲突

## deps 专属禁止输出

主文"共用禁止输出"已覆盖通用项；以下是 deps 子模式补充：

| 禁止输出 | 替代动作 |
|---|---|
| "这个库 star 很多，推荐" | 同时检查维护、release、issue、安全、许可证、项目适配 |
| "我知道有个库" | 联网确认当前状态和官方来源 |
| "直接自己写一个" | 先判断生态是否已有成熟方案和自研成本 |
| "这几个都差不多" | 明确推荐一个，并说明不选其他的原因 |
| "安装后试一下" | 给出具体安装命令、最小接入和验证命令 |
| "应该没安全问题" | 查 audit / CVE / SECURITY.md / OpenSSF 或说明未验证原因 |
