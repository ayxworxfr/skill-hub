# Safe Refactor Workflow

## 目标

在不改变预期行为的前提下降低复杂度。重构必须先保护行为，再改变结构。

## 适用范围

优先用于：

- 用户说“重构”“整理结构”“抽公共逻辑”“代码太乱”
- 拆大函数、大组件、大文件
- 消除重复、收敛条件分支、改善命名
- 迁移目录但不改变功能

不要用于：

- 需求行为本身要变化：用 `general-development-workflow`
- 修 bug：用 `bugfix-and-debug-workflow`
- 大范围架构取舍：先用 `solution-design-workflow`

## 强制流程

### 1. 锁定外部行为

先列出必须保持不变的内容：

- 输入输出
- 异常和错误信息
- 文件格式和字段名
- UI 行为和交互
- 公共 API、props、事件、配置
- 调用方依赖

不知道调用方时，先查调用方。

### 2. 建立保护

优先使用现有测试。没有测试时，至少建立一个可重复验证路径：

- 最小回归测试
- 命令输出对比
- 示例输入输出对比
- UI 手动路径
- 类型或构建检查

### 3. 只做一种主要重构

每轮聚焦一个主目标：

- 拆函数/组件
- 消除重复
- 收敛分支
- 改命名
- 调整目录
- 提取共享逻辑

不要把行为变更混入重构。

### 4. 执行并验证

实施规则：

- 保留模块边界，除非边界本身就是问题
- 删除无用代码，不注释保留
- 不引入过度抽象
- 不扩大公共 API
- 每完成一个重构目标就跑最小验证

## 检查门

以下情况不得继续：

- 没有列出必须保持的外部行为
- 没有调用方或验证路径
- 同时做多个不相关重构
- 重构中顺手改变需求
- 验证失败但继续扩大改动

## 禁止输出模式

| 禁止输出 | 替代动作 |
|---|---|
| “只是整理结构” | 列出保持不变的行为和验证方式 |
| “顺便优化一下逻辑” | 明确这是行为变更，切换开发或调试流程 |
| “抽公共组件更优雅” | 说明复用点、调用方和 API 边界 |
| “没有测试也没关系” | 建立最小可重复验证路径 |
| “改动比较大但应该没影响” | 分段验证并列出影响范围 |

## 最终输出

必须说明：

- 保持不变的行为
- 结构上做了什么整理
- 删除了哪些失效或重复逻辑
- 验证方式和结果
- 未覆盖风险；没有写“无”
