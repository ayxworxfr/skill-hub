# Pytest Workflow

## 目标

测试必须验证行为，不服务于掩盖半成品实现。先覆盖目标行为或失败复现，再修改实现。

## 适用范围

优先用于：

- 新增 pytest 测试
- 修复失败测试
- 为 bug 补回归测试
- 为重构建立行为保护
- 验证 Python 数据处理、文件输出、异常、外部依赖 mock

不要用于非 Python 测试或纯前端测试。

## 强制流程

### 1. 读取现有测试模式

先检查：

- 测试目录和命名
- fixture、factory、mock、参数化风格
- 断言习惯
- 临时文件、环境变量、外部依赖处理方式
- 运行命令

禁止引入仓库没有采用的测试风格，除非现有方式无法覆盖。

### 2. 定义被测行为

每个测试必须写清：

- 输入
- 行为
- 预期输出或副作用
- 失败路径或边界条件

Bug 修复：优先写能失败的回归用例。
重构：先确认旧行为，再改结构。

### 3. 写聚焦测试

规则：

- 一条测试验证一个主要行为
- 测试数据小而完整
- 断言外部可观察结果：返回值、文件、异常、调用参数、输出内容
- 不复制实现逻辑生成期望值
- 不用宽松断言遮住错误

### 4. 运行并处理结果

失败时：

- 读完整断言差异或 traceback
- 判断是测试预期错、fixture 错、实现错还是环境错
- 修正后重新运行同一最小测试
- 通过后按影响范围决定是否跑更大范围

## 检查门

以下情况不得声称测试有效：

- 测试只有 `assert True`
- 测试只覆盖 happy path，但需求有明确失败路径
- 断言复制了实现逻辑
- 为了通过测试删除或放宽关键断言
- 测试绿了但没有覆盖本次需求或 bug

## 禁止输出模式

| 禁止输出 | 替代动作 |
|---|---|
| “补了测试” | 说明测试覆盖的输入、行为和预期 |
| “测试过了” | 给出 pytest 命令和结果 |
| “这个不好测” | 说明不可测原因，并给出替代验证 |
| “先 mock 掉” | 说明 mock 边界和真实行为如何保证 |
| “只测主流程够了” | 对照需求列出未覆盖分支 |

## 输出格式

```markdown
测试：
- 新增/修改：
- 覆盖行为：
- 命令：
- 结果：
- 未覆盖：
```
