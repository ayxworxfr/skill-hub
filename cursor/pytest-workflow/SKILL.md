# Pytest Workflow

## 目标

让测试先服务于行为确认，再服务于代码修改。

## 完整性红线

- 测试是验证手段，不是用来掩盖实现不完整的补丁
- 不能通过只写迎合当前实现的测试，来让半成品逻辑看起来“已完成”
- 不能因为补了测试，就默认代码实现已经足够完整
- 如果需求要求完整行为覆盖，测试也要覆盖关键分支，而不是只验证最容易通过的路径

## 反偷懒执行规则

- 不只跑同一个失败用例反复碰运气；要根据报错和断言结果调整验证策略
- 不把“测试环境问题”当默认借口；先检查 fixture、mock、输入数据、调用路径
- 不闲置已有测试和最小验证手段；能加回归测试就不要只口头说“应该没问题”
- 不通过删断言、放宽断言、只测 happy path 来制造虚假通过
- 不在测试刚变绿就停止思考；要确认测试是否真正覆盖了本次需求或修复点

## 工作流

### 1. 先看已有测试风格

- 优先复用现有 fixture、mock、参数化方式
- 保持测试文件命名、函数命名、断言风格一致
- 不引入这个仓库本来没有的测试框架写法

### 2. 先覆盖问题场景

- 有 bug：先补回归测试
- 有新功能：先补目标行为测试
- 有重构：先补保护性测试，再改实现

### 3. 只测对外行为

- 优先断言返回值、生成文件、数据内容、异常、调用参数
- 少测内部实现细节
- 仅在必要时断言 mock 调用次数和入参

### 4. 保持测试可读

- 测试数据尽量小而完整
- 一条测试只验证一个主要行为
- 名称直接表达输入和预期

### 5. 修改后验证

- 优先运行最小相关测试
- 如果改动影响多个分支，再补边界测试
- 如果实现仍是半成品，不能用“测试已通过”作为交付完成的理由

## 常见断言优先级

1. 业务结果是否正确
2. 文件或输出是否生成
3. 字段名和列顺序是否正确
4. 异常类型和错误信息是否正确
5. 外部依赖调用是否符合预期

## 避免事项

- 不写只有 `assert True` 的测试
- 不堆很多无关断言到一个测试里
- 不为了让测试通过而复制实现逻辑
- 不跳过关键失败场景
- 不用测试去合理化缩水实现
- 不只测 happy path 就宣称需求已覆盖
- 不把“测试绿了”偷换成“实现完整了”

## 适用信号

- 用户提到 `pytest`
- 需要补测试或修测试
- 需要做回归验证
- 改动涉及文件输出、DataFrame、异常处理、外部依赖 mock
