# Reading Code

## Core Principle

把读代码从"随机浏览"改成"证据驱动的结构化理解"。先建地图，再追主路径，最后标影响范围。每一条结论都要能指回具体文件/函数/行号；不能就写"未确认"。

## 适用范围

优先用于：

- 梳理模块、解释文件、理解大文件
- 追调用链、数据流、状态流、副作用
- 改代码前评估影响范围
- 旧模块、复杂组件、历史包袱

不要用于：

- 找风险/判断能否合并：用 `reviewing-code`
- 已经理解，要修 bug：用 `debugging`
- 已经理解，要实现新功能：用 `building`

## 强制流程

### 1. 确定理解目标

判断用户关心哪类问题（不止一类时按顺序处理）：

- 模块职责
- 调用链
- 数据流
- 状态流
- 影响范围
- 大文件拆解

没说清时按"职责 → 主路径 → 数据流 → 影响范围"收敛。

### 2. 建结构地图

必须列出（每项至少 1 条证据：文件路径或函数名）：

- **入口**：谁调用它，用户从哪里进入
- **核心单元**：类、函数、组件、hook、服务、配置
- **输出/副作用**：返回值、文件、网络、UI、数据库、状态更新
- **外部依赖**：配置、环境变量、第三方库、API、事件

### 3. 追主路径

沿主路径列出：

- 输入从哪里来
- 中间经过哪些转换（每步标文件:函数）
- 分支条件是什么
- 最终输出到哪里
- 失败路径如何处理

数据流问题必须标到字段；调用链问题必须标入口和落点。

### 4. 中段自检

进入"标影响范围"前必须确认：

- [ ] 主路径每一步都有具体文件:函数证据，没有"应该"/"大概"
- [ ] 主路径和支线有显式区分，没有把支线当主路径
- [ ] 关键分支的条件已写清，不是"根据情况"

任一未通过 → 回 Step 3 补证据，禁止前进。

### 5. 标影响范围

列出（每项必须给出搜索方式或调用方位置）：

- **直接调用方**
- **间接受影响模块**（通过类型/接口/事件传递）
- **相关测试**
- **相关配置/类型/文档**
- **改动风险和未知点**

### 6. Final Gate

输出前逐项确认：

- [ ] 整体职责一句话能说清
- [ ] 结构地图四项齐全（入口/核心/输出/依赖）
- [ ] 主路径有完整证据链
- [ ] 影响范围包括"未确认"项，没有用"应该影响不大"打发
- [ ] 所有结论都能指回具体文件:函数

任一项未通过不得输出。

## 检查门

以下情况不得下结论：

- 只读了一个文件，没看入口或调用方
- 只复述函数名，没解释职责
- 没区分主路径和支线
- 没说明证据来源
- 影响范围没有测试、配置或调用方视角

## 禁止输出模式

| 禁止输出 | 替代动作 |
|---|---|
| "这个模块主要是处理数据" | 说明输入、转换、输出、副作用，每项有文件:函数 |
| "调用比较复杂" | 画出关键调用链和分支点 |
| "影响不大" | 列出检查过的调用方和未检查范围 |
| "大概是这里" | 给出文件:函数证据，并标记不确定点 |
| "代码有点乱" | 按职责拆区块，指出具体耦合点 |
| "应该是 X 用到的" | 用搜索/grep 确认调用方，再下结论 |

## 输出格式

```markdown
## 整体职责

[一句话说明模块/文件做什么]

## 结构地图

- 入口：<文件:函数>
- 核心单元：<列出>
- 输出/副作用：<列出>
- 外部依赖：<列出>

## 主路径

1. <文件:函数> → <做什么>
2. ...

## 数据流或调用链

- <字段/参数> 从 <源> 到 <终点>，经过 <转换>

## 影响范围

- 直接影响：<调用方列表>
- 间接影响：<列出>
- 相关验证：<测试/配置位置>
- 未确认：<明确列出，不写"无">
```
