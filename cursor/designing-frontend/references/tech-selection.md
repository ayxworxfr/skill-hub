# Tech Selection

## 核心原则

前端技术选型先服务产品和团队约束，再服务个人偏好。默认复用项目已有栈，除非现有栈无法满足关键需求。

## 决策流程

1. 明确任务：
   - 页面应用、组件库、营销页、后台系统、实时协作、数据看板？
2. 识别约束：
   - SEO、SSR、离线、权限、性能、团队熟悉度、部署平台。
3. 盘点现有能力：
   - 项目已经有什么框架、组件库、样式系统、状态管理？
4. 比较候选：
   - 只保留 2-3 个真实可落地方案。
5. 给出推荐：
   - 说明为什么不是其他方案。
6. 定义退出条件：
   - 什么时候该回滚或换方案？

## 常见选择

| 问题 | 默认建议 |
|---|---|
| 普通 SPA 工具应用 | Vite + React/Vue |
| 需要 SEO/SSR/路由约定 | Next 或项目已有 SSR 框架 |
| 小型局部交互 | 不引入新框架，使用现有栈 |
| 简单局部状态 | React state / reducer |
| 服务端数据缓存 | TanStack Query / SWR / 项目已有请求层 |
| 复杂跨页面客户端状态 | Zustand/Redux 等项目已有方案 |
| 样式 | 优先项目已有 Tailwind/CSS Modules/token |
| 复杂无障碍基础组件 | Radix/Headless UI/项目已有 UI 层 |

## 新依赖检查门

新增依赖前必须回答：

- 当前项目是否已有等价能力？
- 原生 Web/CSS 能否解决？
- 新依赖包体和 tree-shaking 如何？
- SSR/CSR 是否兼容？
- 类型质量如何？
- 可访问性质量如何？
- 样式主题能否接入项目 token？
- 未来迁移成本如何？

## 禁止模式

- 为一个按钮、弹窗或简单动效引入大型库。
- 因为“更现代”替换项目已有稳定栈。
- 绕过项目 UI 层直接散用第三方组件。
- 不看 `package.json` 就推荐新技术。
- 只列流行度，不比较当前项目约束。
