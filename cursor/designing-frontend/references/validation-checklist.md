# Validation Checklist

## 目标

前端交付必须有可验证结论。不能用“看起来可以”“应该没问题”代替检查。

## 验证选择

根据改动类型选择最小有效验证：

| 改动类型 | 必须优先验证 |
|---|---|
| TypeScript/组件 props | typecheck |
| 样式和布局 | 浏览器检查、截图、Storybook、Playwright |
| 业务交互 | 相关测试或手动主流程 |
| 路由/构建配置 | build |
| 可访问性 | 键盘检查、axe、手动 focus 检查 |
| 性能相关 | bundle、render 次数、网络请求、列表规模 |

## 验证报告格式

```text
验证：
- 命令：
- 结果：
- 浏览器检查：
- 响应式检查：
- 未验证：
- 风险：
```

## UI 手动检查路径

至少检查：

- 默认状态。
- loading 状态。
- empty 状态。
- error 状态。
- 长文本/长列表/大图片/代码块。
- 窄屏。
- 键盘操作。

## 复杂流式界面验证

检查以下内容类型：

- 单行内容。
- 长文本内容。
- Markdown 或富文本内容。
- 代码块。
- 流式更新内容。
- error/retry 状态。
- 连续同类内容。
- 320px 窄屏。

预期：

- 内容只出现在声明的内容槽。
- 装饰、标识、操作区不被主内容侵占。
- 长内容不进入固定槽位。
- 代码块在内容槽内横向滚动。
- 输入区或主操作区稳定，不被内容列表挤走。

## 常见命令发现方式

优先读项目已有配置：

- `package.json` scripts。
- `Makefile`。
- CI 配置。
- README 开发说明。

不要编造命令。找不到命令时，说明“未验证：仓库未提供对应脚本或未找到脚本”。

## 阻断条件

- 没跑验证却写“已验证”。
- 只跑无关命令证明 UI 改动正确。
- 浏览器不可用时不说明剩余视觉风险。
- 构建失败后仍声称完成。
