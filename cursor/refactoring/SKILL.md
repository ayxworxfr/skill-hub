# Refactoring

## Core Principle

在不改变预期行为的前提下降低复杂度。重构必须先定义行为签名、建立保护，再小步改变结构并逐步验证。**禁止在重构中夹带需求变更；禁止"顺便优化"；禁止没有验证路径就改结构。**

## 适用范围

优先用于：

- 用户说"重构"/"整理结构"/"抽公共逻辑"/"代码太乱"
- 拆大函数、大组件、大文件
- 消除重复、收敛条件分支、改善命名
- 迁移目录、移动模块、拆分模块但不改功能
- 降低耦合、改善依赖方向、收敛公共 API

不要用于：

- 需求行为本身要变化：用 `building`
- 修 bug：用 `debugging`
- 大范围架构取舍：先用 `planning`
- 前端布局/组件设计重做：叠加 `designing-frontend`

## 强制流程

### 1. 定义行为签名

重构前必须列出必须保持不变的外部行为：

- 输入和输出
- 异常类型、错误信息、状态码
- 文件格式、字段名、排序、单位、时区、精度
- UI 行为、焦点、交互、响应式表现
- 公共 API、props、事件、配置项、CLI 参数
- 数据库、网络、文件、副作用
- 调用方依赖和测试入口

不知道调用方时，先用搜索和调用链分析补齐（参考 `reading-code` 思路）。

### 2. 建立基线保护

改结构前至少建立一种可重复验证：

- 现有测试命令
- 最小回归测试
- golden input/output 对比
- CLI 或脚本输出对比
- UI 手动路径或截图对比
- typecheck / build / import check

**高风险重构没有验证路径时，先补保护；不能直接改。**

### 3. 风险分级

按风险决定步幅：

| 风险 | 场景 | 策略 |
|---|---|---|
| 低 | 局部命名、提取私有辅助函数 | 一次完成，跑局部验证 |
| 中 | 拆文件、移动模块、消除重复分支 | 分阶段改，每阶段跑验证 |
| 高 | 公共 API、跨模块迁移、状态机、数据格式 | 先 planning，必要时分支/worktree，逐步迁移 |

**高风险重构不得和功能变更混在同一次交付。**

### 4. 中段自检

进入实施前确认：

- [ ] 行为签名已写出，至少 3 项可验证
- [ ] 至少一种基线验证路径已建立或确认存在
- [ ] 风险分级判断正确，没有把高风险当低风险
- [ ] 主重构目标只有一个

任一未通过 → 回 Step 1-3 补，禁止前进。

### 5. 只做一个主重构目标

每轮聚焦一个主目标：

- 拆函数/组件
- 消除重复
- 收敛条件分支
- 改命名
- 移动模块
- 提取共享逻辑
- 降低依赖耦合

**发现行为 bug 时，暂停重构，切换 `debugging`。发现需求变化，停下切换 `building`。**

### 6. 小步实施

实施规则：

- 先移动/拆分，再改内部实现；不要同时换行为和换位置
- 保留模块边界，除非边界本身就是问题
- 公共 API 变更必须检查所有调用方
- 删除无用代码，不注释保留废代码
- 不新增兼容层，除非这是已发布接口或持久化数据的必要迁移
- 不为了"更优雅"引入多层抽象

### 7. 每阶段验证

每完成一个阶段必须验证：

- 行为签名是否仍成立
- 调用方是否仍能导入/调用
- 类型、构建、测试是否通过
- 生成文件、快照、输出是否符合预期

详细策略见 `verifying` skill。验证失败时停止扩大改动，先定位失败属于行为变化、测试脆弱还是导入路径错误。

### 8. Final Gate

输出前确认：

- [ ] 行为签名仍成立（逐项核对）
- [ ] 基线验证全部通过
- [ ] 删除/迁移的旧逻辑已显式列出
- [ ] 调用方/配置/导入路径都已检查
- [ ] 没有触发"禁止输出模式"任一行
- [ ] 没有夹带需求变更或顺手优化

任一未通过不得声称完成。

## 特殊重构门

### 公共 API / Props / 配置

必须：

- 找所有调用方（搜索 + 调用链分析）
- 确认是否稳定接口
- 更新类型、文档、测试
- 验证旧调用路径是否被移除或迁移完成

### 移动文件 / 改目录

必须：

- 更新 import/export
- 更新测试、文档、配置、构建别名
- 检查动态 import、字符串路径、注册表、入口文件
- 运行导入或构建验证

### 抽共享逻辑

必须：

- 至少两个真实调用方受益
- 新抽象名字表达业务或技术职责
- 不把业务差异塞进布尔参数泥潭
- 调用方读起来比原来更清楚

## 检查门

以下情况不得继续：

- 没有行为签名
- 没有基线验证路径
- 同时做多个不相关重构
- 重构中顺手改变需求
- 公共 API 改动未查调用方
- 移动文件后未检查导入、配置和测试
- 验证失败但继续扩大改动

## 禁止输出模式

| 禁止输出 | 替代动作 |
|---|---|
| "只是整理结构" | 列出行为签名和验证方式 |
| "顺便优化一下逻辑" | 明确这是行为变更，切换 `building` 或 `debugging` |
| "抽公共组件更优雅" | 说明复用点、调用方和 API 边界 |
| "没有测试也没关系" | 建立最小可重复验证路径 |
| "改动比较大但应该没影响" | 分阶段验证并列出影响范围 |
| "先保留兼容旧写法" | 说明是否为已发布接口；否则直接迁移并删除旧逻辑 |
| "拆完看起来更清楚" | 给出拆前/拆后的职责对比和调用方影响 |

## 最终输出

```markdown
## 行为签名

- <列出必须保持不变的项>

## 主重构目标

<本轮一句话说明>

## 结构变化

- 拆/移/抽/改名：<具体动作>

## 删除/迁移的旧逻辑

- <列出，没有写"无">

## 调用方/配置/导入路径检查

- <列出已检查的范围>

## 验证

- 基线对照：<改动前后行为对照>
- 命令：
- 结果：

## 风险

- <未覆盖路径或剩余风险，没有写"无">
```
