# General Development Workflow

## 目标

把普通开发任务收敛为可验证交付：先定义目标和边界，再读现有模式，然后做最小完整实现，最后用证据证明结果。

## 适用范围

优先用于：

- 新增或修改业务逻辑、脚本、工具函数、模块能力
- 用户说“实现”“改一下”“补上这个能力”“按现有风格改”
- 需求可以在当前轮次内完成，不需要长期里程碑管理

不要用于：

- 报错、异常结果、根因排查：用 `bugfix-and-debug-workflow`
- 只要方案或架构取舍：用 `solution-design-workflow`
- 只做行为不变的结构整理：用 `safe-refactor-workflow`
- 前端 UI、布局、组件设计：用 `frontend-product-design-workflow`
- 代码审查：用 `code-review-workflow`

## 强制流程

### 1. 定义任务契约

开始编码前写清：

- 目标：要改变什么行为或新增什么能力
- 非目标：明确不处理哪些相邻问题
- 输入：数据、参数、文件、接口、用户操作
- 输出：返回值、文件、UI、状态、日志、副作用
- 验收：用什么命令、测试、对比或手动路径证明完成

缺少会影响实现判断的关键约束时，先补齐；不要边写边猜。

### 2. 读取现有模式

至少检查：

- 入口文件和直接调用方
- 被改逻辑的上下游数据流
- 相邻测试、配置、类型定义、错误处理方式
- 项目已有同类实现、命名、目录和工具函数

禁止只看一个函数就改公共行为。

### 3. 设计最小完整改动

改动必须满足：

- 覆盖任务契约里的关键分支
- 删除已失效旧逻辑，不注释保留废代码
- 不新增临时兼容、占位分支、默认值掩盖问题
- 不为未来假设做抽象
- 不顺手重构无关模块

如果发现需求实际是 bug、重构、配置或前端问题，切换对应 Skill。

### 4. 实施

实施规则：

- 跟随项目现有代码风格和目录结构
- 保持改动面与任务直接相关
- 新注释只解释不明显的业务约束或关键判断
- 不写占位函数、空分支、伪实现
- 不为了少改文件而牺牲正确性

### 5. 验证

完成后必须使用 `verification-workflow` 思路选择最小有效验证：

- 相关测试
- lint/typecheck
- build
- 运行脚本
- 手动复现路径
- 文件/输出对比

无法验证时，写清原因、已检查内容和剩余风险。

## 检查门

未满足以下条件不得声称完成：

- 已读到现有模式，而不是凭通用经验实现
- 已覆盖主要输入、输出、失败路径和边界条件
- 已删除或适配过时逻辑
- 已运行最小相关验证，或明确说明无法验证

## 禁止输出模式

| 禁止输出 | 替代动作 |
|---|---|
| “先做一个简单版” | 交付完整实现，或明确说明阻塞缺口 |
| “常见场景可以用了” | 对照验收标准列出覆盖和未覆盖路径 |
| “其他保持不变” | 明确列出未修改范围或让 diff 自证 |
| “应该没问题” | 给出验证命令、结果和剩余风险 |
| “这里加个兜底” | 追溯为什么数据/状态会异常并修根因 |

## 最终输出

按需包含：

- 结果：改了什么行为
- 依据：为什么这样改，复用了哪些项目模式
- 验证：具体命令或检查路径和结果
- 残留：未验证项或风险；没有则写“无”
